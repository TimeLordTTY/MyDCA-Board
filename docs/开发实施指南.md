# 财富中枢系统 - 完整开发实施指南

**作者**：timelordtty

## 文档说明

本文档是《财富中枢系统完整设计方案.md》的实施指南，提供从零开始构建完整系统的详细步骤、验证方法和开发顺序。

**技术栈**：
- 后端：Java 17+ (Spring Boot 3.x) + MyBatis（不使用JPA）
- 前端：Vue 3 + TypeScript（PC: Element Plus/Naive UI, Mobile: Vant）
- Python：3.9+（行情、指标、策略、回测）
- 数据库：MySQL 8.0（已执行DDL脚本）
- 部署：Nginx（静态资源 + 反向代理），无Docker

**关键配置**：
- 包名：`com.timelordtty.dca`
- 接口版本：`/api/v2/`（v1为旧版本，当前为v2重构版本）
- 服务端口：8765
- 数据库：已配置并执行DDL

**开发原则**：
- **Phase 1专注后端开发**，完成所有后端API和业务逻辑
- **Phase 2开始前端开发**，同时进行行情数据集成
- 每个阶段完成后进行验证，验证通过才能进入下一阶段
- 严格遵循设计方案，不引入中间件（无MQ、Redis、ES等）
- 本文档提供开发步骤和验证方法，不包含完整代码实现

#### 步骤1.1.4：实现JWT认证服务

**文件位置**：`backend/src/main/java/com/timelordtty/dca/security/`

**需要创建的类**：
1. `JwtTokenProvider.java` - JWT Token生成和验证
   - 方法：generateToken, getUsernameFromToken, validateToken
   - 配置：secret（密钥），expiration（过期时间，默认24小时）
2. `JwtAuthenticationFilter.java` - JWT认证过滤器
   - 从请求头`Authorization: Bearer <token>`提取token
   - 验证token并设置SecurityContext
3. `SecurityConfig.java` - Spring Security配置
   - 配置密码加密器（BCryptPasswordEncoder）
   - 配置SecurityFilterChain：公开`/api/auth/**`，其他接口需要认证
   - 添加JwtAuthenticationFilter到过滤器链

**关键点**：
- JWT secret需要在配置文件中设置（生产环境必须修改）
- 使用Spring Security 6.x的新API（`requestMatchers`）
- Session策略设置为STATELESS（无状态）

**验证要点**：
- POST `/api/v2/auth/register` - 注册成功，返回token和用户信息
- POST `/api/v2/auth/login` - 登录成功，返回JWT token
- GET `/api/v2/users/me` - 使用Bearer token访问成功，返回当前用户信息
- 无token访问返回401 Unauthorized

#### 步骤1.2.2：创建账户实体与Mapper

**文件位置**：
- Model: `backend/src/main/java/com/timelordtty/dca/model/Account.java`
- Mapper: `backend/src/main/java/com/timelordtty/dca/mapper/AccountMapper.java`
- XML: `backend/src/main/resources/mapper/AccountMapper.xml`

**Account实体字段**：
- id, accountCode, accountName, accountType（CASH/POSITION/INCOME/EXPENSE/FEE/RECEIVABLE/LIABILITY）, accountKind（REAL/VIRTUAL）, virtualSubtype（仅VIRTUAL账户）, ownerUserId, ownerFamilyId, ownerType（PERSONAL/SHARED）, parentAccountId（父子账户关系）, fundUsage（SPENDABLE/RESERVED/INVESTABLE，仅REAL CASH叶子账户）, initialBalance, balance, reservedAmount, note, createdAt, updatedAt

**AccountMapper方法**：
- selectById, selectByOwner（按userId和familyId查询）, selectChildren（查询子账户）, selectLeafAccounts（查询叶子账户，用于记账校验）, insert, update, updateBalance, updateReservedAmount

**关键点**：
- `parentAccountId`用于父子账户关系（父账户=平台容器，子账户=真实信封余额）
- `fundUsage`仅对REAL CASH叶子账户生效
- 需要支持查询子账户和叶子账户

#### 步骤1.2.3：实现账户服务（应用层校验）

**文件位置**：`backend/src/main/java/com/timelordtty/dca/service/AccountService.java`

**关键业务规则（应用层必须校验）**：
1. VIRTUAL账户不允许设置parent_account_id
2. 子账户必须是REAL
3. 只有CASH类型的REAL账户允许形成父子层级
4. `ledger_posting.account_id`只允许引用叶子账户（禁止对父账户记账）

**AccountService方法**：
- createAccount（创建账户，包含上述校验）
- updateAccount（更新账户信息）
- getAccountTree（获取账户树形结构，父账户余额=Σ子账户余额）
- calculateParentBalance（计算父账户余额，聚合子账户）
- isLeafAccount（判断是否为叶子账户，用于记账校验）
- adjustBalance（调整余额，生成ADJUST类型流水）

**关键点**：
- 父账户不参与任何记账分录，禁止余额编辑
- 父账户展示余额 = Σ(子账户叶子余额)（仅展示层聚合）
- 余额调整必须生成ADJUST类型的ledger_txn和ledger_posting

**验证要点**：
- 创建VIRTUAL账户时设置parent_account_id应返回400错误
- 创建非CASH类型的REAL账户作为子账户应返回400错误
- 父账户余额 = 所有子账户余额之和
- 尝试对父账户创建ledger_posting应返回400错误

#### 步骤1.2.4：实现产品与账户管理Controller

**文件位置**：
- `backend/src/main/java/com/timelordtty/dca/controller/ProductController.java`
- `backend/src/main/java/com/timelordtty/dca/controller/AccountController.java`

**ProductController接口**：
- GET `/api/v2/products` - 获取产品列表（支持keyword、assetType、channel筛选）
- GET `/api/v2/products/{id}` - 获取产品详情
- POST `/api/v2/products` - 创建产品
- PUT `/api/v2/products/{id}` - 更新产品

**AccountController接口**：
- GET `/api/v2/accounts` - 获取账户列表（支持ownerType筛选，返回树形结构）
- GET `/api/v2/accounts/{id}` - 获取账户详情（父账户包含子账户列表和聚合余额）
- POST `/api/v2/accounts` - 创建账户（包含应用层校验）
- PUT `/api/v2/accounts/{id}` - 更新账户
- PUT `/api/v2/accounts/{id}/balance` - 调整账户余额（生成ADJUST流水）

**关键点**：
- 账户列表接口需要支持树形结构返回（父账户包含children数组）
- 父账户的balance字段在返回时计算为子账户余额之和
- 余额调整接口必须生成ADJUST类型的流水，不允许直接修改balance字段

**验证要点**：
- 产品列表筛选功能正常
- 账户树形结构正确返回
- 父账户余额计算正确
- 余额调整生成ADJUST流水
- 应用层校验规则生效

### Phase 1.3: 统一记账模块（核心地基）（5-7天）

#### 步骤1.3.1：创建记账实体与Mapper

**文件位置**：
- Model: `backend/src/main/java/com/timelordtty/dca/model/LedgerTxn.java`, `LedgerPosting.java`
- Mapper: `backend/src/main/java/com/timelordtty/dca/mapper/LedgerTxnMapper.java`, `LedgerPostingMapper.java`
- XML: `backend/src/main/resources/mapper/LedgerTxnMapper.xml`, `LedgerPostingMapper.xml`

**LedgerTxn实体字段**：
- id, bizGroupKey（业务分组键，同一业务事件的多条分录共享）, txnType（TRANSFER/BUY/SELL/SUBSCRIPTION/REDEMPTION/CUSTODY_TRANSFER/BOND_REPO/EXPENSE/INCOME/ADJUST/REIMBURSE_IN/REIMBURSE_OUT等）, orderId（关联订单ID）, relatedTxnId（关联的原交易txn_id，用于退款/报销/撤销/转托管）, relatedOrderId（关联的原订单号）, relationType（NONE/TRANSFER_PAIR/REFUND/REFUND_OF/REIMBURSE/REIMBURSEMENT_OF/REVERSAL/CUSTODY_TRANSFER_OF）, tradeDate, navDate, confirmDate, status（PENDING/CONFIRMED/CANCELLED/REVERSED）, subject, note, createdAt, updatedAt

**LedgerPosting实体字段**：
- id, txnId, accountId（必须是叶子账户）, productId（持仓类分录需要）, postingType（DEBIT/CREDIT）, amount（永远为正数）, shares（持仓类分录，永远为正数，NULL或>=0）, nav（净值，确认时）, fee（手续费）, createdAt

**LedgerTxnMapper方法**：

#### 步骤1.3.2：实现记账服务核心逻辑

**文件位置**：`backend/src/main/java/com/timelordtty/dca/service/LedgerService.java`

**LedgerService核心方法**：
- createTransaction（创建交易流水，双分录设计）
  - 生成bizGroupKey（UUID）
  - 创建ledger_txn记录
  - 验证借贷平衡（DEBIT总额 = CREDIT总额）
  - 校验accountId必须是叶子账户
  - 创建ledger_posting记录
  - 更新账户余额（资产类账户：DEBIT增加，CREDIT减少）
- getTransactions（查询流水列表，支持筛选和分页）
- getTransactionDetail（获取流水详情，包含所有postings）
  - 如果原交易有退款/报销，聚合显示refundedTotal、reimbursedTotal、remaining
- reverseTransaction（撤销流水，生成REVERSED类型的反向流水）
  - 设置relatedTxnId指向被撤销的交易
  - 设置relationType='REVERSAL'
- createRefund（创建退款交易）
  - 参数：relatedTxnId（原交易ID）, refundAmount, accountId
  - 设置relationType='REFUND_OF'（推荐）或'REFUND'（兼容）
  - 生成分录：CASH DEBIT + 原交易EXPENSE CREDIT（或按业务规则）
- createReimburse（创建报销交易）
  - 参数：relatedTxnId（原交易ID）, reimburseAmount, accountId
  - 设置relationType='REIMBURSEMENT_OF'（推荐）或'REIMBURSE'（兼容）
  - 生成分录：CASH DEBIT + 原交易EXPENSE CREDIT（或按业务规则）

**借贷方向规则**（必须严格遵循）：
- 资产账户（CASH、POSITION、RECEIVABLE）：DEBIT = 增加，CREDIT = 减少
- 负债账户（LIABILITY）：DEBIT = 减少债务，CREDIT = 增加债务
- 收入账户（INCOME）：DEBIT = 减少，CREDIT = 增加
- 支出账户（EXPENSE、FEE）：DEBIT = 增加，CREDIT = 减少
- 余额公式：`balance = initial_balance + Σ(DEBIT) - Σ(CREDIT)`（资产类账户）

**关键业务规则**：
1. 借贷必须平衡：每笔交易的DEBIT总额 = CREDIT总额
2. `ledger_posting.account_id`只允许引用叶子账户（禁止父账户）
3. `amount`永远为正数，方向由`postingType`决定
4. `shares`永远为正数（NULL或>=0），方向由`postingType`决定
5. 余额更新必须在事务中完成，保证一致性

**验证要点**：
- 转账交易：DEBIT总额 = CREDIT总额
- 消费交易：CASH CREDIT + EXPENSE DEBIT，余额正确减少
- 收入交易：CASH DEBIT + INCOME CREDIT，余额正确增加
- 尝试使用父账户ID创建posting应返回400错误
- 借贷不平衡应返回400错误

#### 步骤1.4.2：实现订单服务

**文件位置**：`backend/src/main/java/com/timelordtty/dca/service/OrderService.java`

**OrderService核心方法**：
- createOrder（创建订单，支持组合支付）
  - 参数：fundingLines（数组，每个元素包含accountId和amount）, productId, orderType
  - 校验：Σ(fundingLines.amount) = requestedAmount（应用层校验）
  - 校验每个账户可用余额：`account.balance - account.reserved_amount >= fundingLine.amount`
  - 校验账户fund_usage（应用层校验）：
    - **EXPENSE订单**：source_account.fund_usage必须=SPENDABLE（专款禁止消费）
    - **BUY/SUBSCRIPTION订单（非逆回购）**：source_account.fund_usage必须=INVESTABLE（除非显式确认放开SPENDABLE）
    - **BOND_REPO订单（逆回购）**：忽略fund_usage限制，允许SPENDABLE/RESERVED/INVESTABLE（逆回购是现金增强工具，不是风险投资）
    - 但仍要严格禁止：RESERVED账户用于日常EXPENSE消费，RESERVED账户用于ETF/股票等风险投资
  - 创建订单记录（status=PENDING）
  - 写入order_funding_line表（每个fundingLine一行）
  - 分别增加各账户的`reserved_amount`（占用资金，不扣款，不生成流水）
  - **逆回购特殊处理**：如果product.asset_type=BOND_REPO，订单类型通过product判定（或新增order_type='BOND_REPO'），下单时只增加reserved_amount，不生成流水
- cancelOrder（取消订单）
  - 只能取消PENDING状态的订单
  - 查询order_funding_line，逐条释放各账户的`reserved_amount`
  - 删除order_funding_line记录（CASCADE自动删除）
  - 更新订单状态为CANCELLED
- getPendingOrders（获取待结算清单）
  - 查询status=PENDING的订单
  - 按expectedConfirmDate排序
  - 关联查询order_funding_line，返回资金来源明细

**关键业务规则**：
1. 订单创建时只增加`reserved_amount`，不生成真实流水
2. `reserved_amount`增加时，`balance`不变
3. 可用购买力 = `balance - reserved_amount`
4. 投资类订单（BUY/SUBSCRIPTION，非逆回购）默认从INVESTABLE账户，如需从SPENDABLE账户需要显式确认
5. **逆回购订单（BOND_REPO）**：
   - 允许从任意fund_usage账户（SPENDABLE/RESERVED/INVESTABLE）作为资金来源
   - 下单时只增加reserved_amount，不生成流水
   - 到期确认时：释放reserved_amount，生成利息流水（CASH DEBIT + INCOME CREDIT）
   - 不生成持仓POSITION（逆回购是现金增强工具，不是投资）

**验证要点**：
- 创建订单后，`reserved_amount`增加，`balance`不变
- 可用余额不足时创建订单应返回400错误
- 取消订单后，`reserved_amount`减少
- 待结算清单正确显示PENDING订单

### Phase 1.5: 持仓计算模块（3-5天）

#### 步骤1.5.1：实现持仓计算服务

**文件位置**：`backend/src/main/java/com/timelordtty/dca/service/HoldingService.java`

**HoldingService核心方法**：
- calculateHoldings（计算实时持仓）
  - 查询所有持仓类分录（account_type=POSITION）
  - 按product_id分组汇总
  - 计算总份额、总成本、平均成本
  - 获取最新净值计算市值和未实现盈亏
- calculateHoldingByProduct（计算单个产品持仓）
- getHoldingHistory（获取持仓历史曲线）

**持仓计算公式**：
- 总份额 = Σ(买入份额DEBIT) - Σ(卖出份额CREDIT)
- 总成本 = Σ(买入成本DEBIT) - Σ(卖出成本扣减CREDIT)
- 平均成本 = 总成本 / 总份额
- 市值 = 总份额 × 最新净值
- 未实现盈亏 = 市值 - 总成本

**成本方法**（MVP默认平均成本法）：
- 买入：增加份额和成本
- 卖出：按平均成本计算成本扣减 = 卖出份额 × 平均成本

**关键点**：
- 持仓计算基于`ledger_posting`实时计算，不依赖快照
- 支持按用户、按产品查询
- 需要从market_nav表获取最新净值（Phase 2实现）

**关键点**：
- jobId唯一约束
- status状态机：PENDING → RUNNING → SUCCESS/FAILED
- 支持进度跟踪（progress）

#### 步骤3.4.2：实现异步任务服务

**文件位置**：`backend/src/main/java/com/timelordtty/dca/service/AsyncJobService.java`

**AsyncJobService核心方法**：
- createJob（创建任务）
  - 生成jobId
  - 创建py_job记录（status=PENDING）
  - 提交到线程池执行
- executeJob（执行任务）
  - 更新status=RUNNING
  - 使用ProcessBuilder执行Python脚本
  - 解析输出，更新outputJson和artifactPath
  - 更新status=SUCCESS/FAILED
- getJobStatus（获取任务状态）
- cancelJob（取消任务）

**关键点**：
- 使用ThreadPoolExecutor管理异步任务
- Python脚本输出写入outputJson
- 大文件（如回测结果）存储在artifactPath

#### 步骤3.4.3：实现异步任务Controller

**文件位置**：`backend/src/main/java/com/timelordtty/dca/controller/JobController.java`

**JobController接口**：
- GET `/api/v2/jobs/{jobId}` - 获取任务状态和进度
- GET `/api/v2/jobs` - 获取任务列表（支持jobType、status筛选）
- POST `/api/v2/jobs/{jobId}/cancel` - 取消任务

2. Mobile端构建：`cd web/mobile-app && npm run build`
3. 部署到Nginx静态目录

**关键点**：
- 构建产物：`dist/`目录
- 配置Nginx静态资源路径

#### 步骤D.3：Nginx配置

**配置文件**：`/etc/nginx/conf.d/wealth-hub.conf`

**配置内容**：
```nginx
server {
    listen 80;
    server_name your-domain.com;

    # PC端静态资源
    location / {
        root /var/www/wealth-hub/pc-app/dist;
        try_files $uri $uri/ /index.html;
    }

    # Mobile端静态资源
    location /m {
        alias /var/www/wealth-hub/mobile-app/dist;
        try_files $uri $uri/ /m/index.html;
    }

    # API反向代理
    location /api {
        proxy_pass http://localhost:8765;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

**关键点**：
- PC端：`/` → PC静态资源
- Mobile端：`/m` → Mobile静态资源
- API：`/api` → Java后端（端口8765）
