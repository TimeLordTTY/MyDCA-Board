# 财富中枢系统完整设计方案

> 当前代码实现基线（2026-05-19）：后端已落地 Spring Boot 3.2 + Java 17 + MyBatis + Spring Security/JWT，接口统一为 `/api/v2/**`；前端为 Vue 3 + TypeScript 双端应用，PC 端使用 Element Plus，Mobile 端使用 Vant；行情/指标由 Java 定时任务调用 Python 脚本落库。当前已实现 Phase 1 后端地基与 Phase 2 前端/行情/指标/定时任务高优先级闭环（18 个 Model、18 个 Mapper、18 个 Service、15 个 Controller；初始化 DDL 当前为 30 张表），Phase 2 仍待补齐流水统计、债券行情、BOLL/KDJ 等增强项；策略建议、复盘导出、回测实验室、异步任务管理属于后续阶段。本文后续设计内容中如出现规划接口或规划表，以本基线和“核心 REST 接口清单（当前实现版）”为准。

## 2. 领域模型与模块拆分

### 2.1 核心领域模型

#### 2.1.1 用户与权限域
- **用户 (User)**: 系统使用者，属于某个家庭
- **家庭 (Family)**: 多用户组织单元，可共享账户和数据
- **角色 (Role)**: ADMIN（管理员，可以管理家庭所有账户和数据）/ MEMBER（普通成员，只能查看自己的数据）
- **权限规则**（权限，即访问和操作数据的权利）: 管理员可看全家，成员只看自己+授权部分

#### 2.1.2 产品域
- **产品主数据 (ProductMaster)**: 全局唯一产品（code + channel + market）
- **用户产品配置 (UserProduct)**: 个人对产品的配置（别名、策略绑定、风险标签等）
- **产品类别**: ETF（交易型开放式指数基金，可在场内像股票一样交易） / LOF（上市型开放式基金，可在场内和场外交易） / FUND（普通基金，场外基金） / MMF（货币基金，类似余额宝） / BANK_WM_NAV（银行理财净值型，按净值计算收益） / BANK_WM_BOX（银行理财预期收益型，预期收益率） / STOCK（股票） / FUTURES（期货） / OPTIONS（期权） / BOND_REPO（国债逆回购，场内交易标的，产品代码如204001/131810等，作为现金管理工具，默认1天期，持有期短、收益以利息体现） 

#### 2.1.3 账户域
- **账户 (Account)**: 资金账户（银行/支付/券商/现金等）或虚拟科目（持仓/费用/收入/负债等）
- **账户分类**: 
  - **REAL（现实账户）**: 银行/支付/券商/现金等，真实存在的账户（如银行卡、支付宝），允许手工余额调整（需生成流水）
  - **VIRTUAL（虚拟科目）**: 持仓/费用/收入/支出/应收/负债等，用于记账的虚拟账户，禁止手工改余额，仅用于分录归集与报表
- **账户归属**: 个人账户（PERSONAL，属于个人所有） / 家庭共同账户（FAMILY，属于家庭共同所有）
- **父子账户语义（资金分区/信封系统）**:
  - **父账户（ROOT）** = 平台容器/分组节点（余利宝、稳利宝、华宝证券等），用于UI聚合与筛选
  - **子账户（BUCKET/叶子账户）** = 资金信封（生活费/房租/项目/理财金/安全金/待分配等），真实金额只存在子账户
  - **父账户余额表述（关键）**：
    - 父账户不参与任何记账分录，禁止余额编辑，balance字段不作为真实余额来源（保持0或不可编辑）
    - 父账户展示余额 = Σ(子账户叶子余额)（仅展示层聚合）
  - **ledger_posting.account_id** 只允许引用【叶子账户】（即不存在任何child的账户）。父账户禁止出现在任何记账分录中（避免把"分组节点"当资金池导致双算/不可追溯）
  - **叶子账户判定规则**：叶子账户=不存在任何child的账户；只有叶子账户允许出现在ledger_posting.account_id
  - **父账户UI行为**：父账户只展示汇总，不允许点进去做"支出/投资/转账来源"选择
  - **待分配子账户**：每个"平台父账户"必须至少有一个默认子账户：xxx_unallocated（待分配/自由资金）。工资/转入平台的资金默认进入待分配，再由用户分配到生活费/房租/项目/理财金等
  - **fund_usage（资金用途）**：
    - `SPENDABLE`（可支出）：允许日常支出/生活消费
    - `RESERVED`（专款）：房租/项目/安全金等，禁止日常支出和默认禁止投资
    - `INVESTABLE`（可投资）：可用于投资如ETF/逆回购等，默认不用于日常支出
    - **约束范围**：只对`account_kind=REAL`且`account_type=CASH`且为叶子账户的场景做约束校验。对VIRTUAL/POSITION/LIABILITY等账户，fund_usage仅保留默认值但不参与业务校验
- **余额口径**（余额，即账户中的资金总额）:
  - `balance`: 账面余额（由流水推导，即所有交易记录计算得出的余额）
  - `reserved_amount`: 占用/冻结金额（占用，即下单时暂时冻结的资金，还未实际扣款；冻结，即资金被暂时锁定不能使用；下单占用，结算确认后释放）
  - `available_for_trade`: 可用购买力（可用，即可以使用的资金） = balance - reserved_amount
  - `available_for_withdraw`: 可取资金（可取，即可以取出来用的钱） = balance - 未交收卖出款（卖出后还未到账的钱） - reserved_amount

#### 2.1.4 交易域（核心）
- **交易事件 (LedgerTxn)**: 业务事件（买入/卖出/赎回/转账/退款/报销/分红等）
- **分录 (LedgerPosting)**: 对账户的借贷影响（借贷，即会计记账方向，借方表示增加或减少，贷方表示相反；金额/份额/费用）
- **订单 (Order)**: 交易意图（发起时间、预期确认日等）
- **结算确认 (SettlementConfirm)**: 实际结算结果（结算，即交易确认和资金到账的过程；确认，即交易最终确认，资金或份额到账；确认日期、实际净值、实际份额/金额）

#### 2.1.5 持仓域
- **持仓 (Holding)**: 由流水推导，可缓存快照（持仓，即当前持有的资产数量和成本）
- **持仓快照 (HoldingsSnapshot)**: 每日持仓快照（提升查询性能）
- **净资产快照 (NetWorthSnapshot)**: 每日净资产快照（净资产，即总资产减去总负债，真正属于你的钱）

#### 2.1.6 行情域
- **日线行情 (MarketBarDaily)**: 长期保存（日线，即每天的开盘价、收盘价、最高价、最低价等数据）
- **实时行情 (MarketQuoteRealtime)**: 短期缓存（7-30天，实时行情，即当前时刻的价格数据）
- **净值 (Nav)**: 长期保存（净值，即基金每份的价格，反映资产价值）
- **指标 (IndicatorDaily)**: 日更指标（分位、回撤、均线等）

#### 2.1.7 策略域
- **策略配置 (StrategyConfig)**: 策略参数配置（策略，即投资规则，如"价格低于20%分位时买入"）
- **产品策略绑定 (ProductStrategyBind)**: 产品与策略的绑定关系
- **策略状态 (StrategyState)**: 策略运行状态（利润池、峰值等）
- **信号 (Signal)**: 策略生成的原始信号（信号，即策略生成的原始交易提示，如"买入"或"卖出"）
- **建议 (Suggestion)**: 面向用户的建议（建议，即系统根据策略生成的交易建议，去重、过期、状态管理）

#### 2.1.8 基本面域（三层）
- **层1 - 产品标签 (ProductTag)**: 长期白名单、风险标签、核心等级
- **层2 - 组合目标 (PortfolioTarget)**: 目标权重范围（权重，即资产在投资组合中的占比，如30%表示占总资产的30%）、再平衡阈值（再平衡，即调整资产配置，使各资产占比回到目标比例；阈值，即触发条件，如"价格低于1.2元时买入"）
- **层3 - 交易档位 (TradingGear)**: 日/周执行档位开关（档位，即执行强度，如低档、中档、高档，油门大小）

#### 2.1.9 回测实验室域（一等模块）
- **回测计划 (BacktestPlan)**: 实验配置（产品范围、策略版本、参数空间、时间范围、成本法、手续费模型）

### 2.4 Python任务执行机制

#### 2.4.1 任务分类

**同步任务**（Java等待Python返回）：
- 单只标的查询：`python scripts/market/get_quote.py --product_id 5 --output result.json`
- 轻量建议生成：`python scripts/strategy/generate_suggestions.py --user_id 1 --product_ids [1,2,3] --output suggestions.json`
- 指标计算：`python scripts/indicator/calc_percentile.py --product_id 5 --window_days 60 --output indicator.json`

**异步任务**（Java启动后返回job_id，前端轮询）：
- 回测执行：`python scripts/backtest/run_backtest.py --run_id 123 --input input.json --output output.json`
- 批量行情归档：`python scripts/market/batch_sync.py --product_ids [1,2,3] --start_date 2023-01-01 --end_date 2024-01-01`
- 报告导出：`python scripts/report/generate_monthly.py --user_id 1 --month 2024-01 --output report.pdf`

#### 2.4.2 任务执行流程

**同步任务流程**：
```
1. Java接收请求
2. Java通过ProcessBuilder启动Python脚本
3. Java等待进程完成（同步阻塞）
4. Python输出JSON到stdout或文件
5. Java读取结果，解析JSON
6. Java返回结果给前端
```

**异步任务流程**：
```
1. Java接收请求
2. Java创建py_job记录（status='PENDING'）
3. Java通过ProcessBuilder启动Python脚本（后台线程）
4. Java立即返回job_id给前端
5. Python执行任务，更新py_job状态（status='RUNNING' -> 'COMPLETED'）
6. 前端轮询 /api/v2/jobs/{job_id} 获取进度和结果（规划接口，当前未实现）
7. 任务完成后，前端获取结果文件路径或摘要数据
```

#### 2.4.3 Python脚本规范

**输入输出约定**：
- 输入：命令行参数（--input input.json）或环境变量
- 输出：JSON格式（stdout或文件）
- 错误：stderr输出，返回非0退出码

**示例脚本结构**：

```java
public List<Suggestion> generateSuggestionsSync(Long userId, List<Long> productIds) {
    ProcessBuilder pb = new ProcessBuilder(
        "python", 
        "scripts/strategy/generate_suggestions.py",
        "--user_id", userId.toString(),

        // 读取输出文件

    } else {

    }

// 异步任务
public String startBacktestAsync(Long runId, BacktestInput input) {
    // 创建job记录
    PyJob job = new PyJob();
    job.setJobType("BACKTEST");
    job.setStatus("PENDING");
    job.setInputJson(objectMapper.writeValueAsString(input));
    pyJobMapper.insert(job);
    
    // 启动后台线程执行
    executorService.submit(() -> {
        try {
            job.setStatus("RUNNING");
            pyJobMapper.updateById(job);
            
            ProcessBuilder pb = new ProcessBuilder(
                "python",
                "scripts/backtest/run_backtest.py",
                "--run_id", runId.toString(),
                "--input", inputPath,
                "--output", outputPath
            );
            
            Process process = pb.start();
            int exitCode = process.waitFor();
            
            if (exitCode == 0) {
                job.setStatus("COMPLETED");
                job.setOutputJson(readOutputFile(outputPath));
                job.setArtifactPath(outputPath);
            } else {
                job.setStatus("FAILED");
                job.setError(readStderr(process));
            }
        } catch (Exception e) {
            job.setStatus("FAILED");
            job.setError(e.getMessage());
        } finally {
            pyJobMapper.updateById(job);
        }
    });
    
    return job.getJobId();
}
```

#### 4.2.1.1 国债逆回购记账方式

**背景**：华宝证券账户内所有资金都允许做【1天期】国债逆回购（包括RESERVED专款），因为会在需要用钱的至少前一天将资金从华宝取出转回余利宝。目标是把华宝当"统一现金管理池"，每天用逆回购吃利息，但仍不允许专款被消费/被拿去买ETF。

**记账实现（方案1：推荐）**：
采用"占用/释放 + 利息入账"方式，不生成持仓POSITION，最贴合"现金管理"场景。

**下单日（当日买入逆回购，本金占用）**：
- 创建 REPO 订单（PENDING），仅增加 source_account.reserved_amount（锁定本金，体现"今日有多少资金在做逆回购"）
- 不生成 ledger_txn 和 ledger_posting（避免破坏信封余额语义）

**到期日（次交易日，到期回款，含利息）**：
确认订单（CONFIRMED），**必须按以下顺序处理（关键）**：
1. **校验订单到期**：trade_date+term 或 confirm_date==today
2. **释放占用**：accounts.reserved_amount -= principal（先释放，避免可用资金瞬间不一致）
3. **生成利息 ledger_txn**：CASH DEBIT(interest) + INCOME CREDIT(interest)（记到同一个发起子账户，后入账）

**利息归属**：
- 利息默认记回原信封（房租/项目/理财金哪个做的repo，就进哪个）
- 如果未来想做"利息归集到理财金/待分配"，可以作为"可选增强"，但不要现在就扩大设计面

**关键规则（资金来源）**：
- 对 **ETF/基金/股票** 等投资类标的：仍维持"默认只允许 fund_usage=INVESTABLE"
- 对 **逆回购（asset_type=BOND_REPO）**：允许从华宝证券下的任意现金叶子账户作为资金来源（SPENDABLE/RESERVED/INVESTABLE 都允许）
- **解释**：逆回购是现金管理工具，短期占用资金换取利息，不改变"用途分区"的长期属性；用户会提前一天出金到余利宝
- **仍要禁止**：专款被日常消费、专款被拿去买ETF

**看板展示（必须用订单汇总计算）**：
- **repo_locked_amount（逆回购占用资金，今日）**：
  - **计算公式**：`repo_locked_amount = Σ(所有REPO订单的principal，status=LOCKED/PENDING/ACTIVE视定义)`
  - **重要**：必须用订单汇总计算，不能用reserved_amount的一部分
  - reserved_amount作为账户总占用（不用拆），可能同时承载下单占用、转账占用、其他冻结
- **available（可用资金）**：`available = balance - reserved_amount`（总可用仍然正确）
- **逆回购累计利息（本周/本月）**：repo_interest_sum（从ledger_txn汇总，txn_type=INCOME且product_id为逆回购产品）
- **明确**：这些只是展示层聚合，不新增表（从订单/流水汇总得出即可）

**示例**：
- 华宝_房租子账户余额10000，做1天期逆回购10000
- 下单日：华宝_房租.reserved_amount += 10000（锁定）
- 到期日：
  1. 校验到期
  2. 华宝_房租.reserved_amount -= 10000（释放）
  3. 生成利息分录：华宝_房租 CASH DEBIT(10) + INCOME CREDIT(10)（利息10元记回房租信封）
- 看板显示：repo_locked_amount = 10000（从订单汇总），华宝_房租可用 = balance - reserved_amount

#### 4.2.3.1 资金分区/信封系统

**业务背景**：
- 平台真实账户只有一个（如余利宝/稳利宝/券商现金），但用户需要在系统中强行区分用途：
  - 余利宝：生活费、房租预备金、项目资金预备金、理财金预备金等
  - 稳利宝：项目资金预备金、房租预备金、安全金等（但产品可能售罄，未来可能转移到券商）
  - 华宝证券：理财金（可买ETF/逆回购）、房租预备金、项目资金预备金等
- 关键目标：即使现实平台不区分，系统也必须做到：
  1) 每笔支出明确"来自哪个分区"，避免误花房租/项目/安全金
  2) 分区资金可在非交易日暂存于余利宝，交易日再整体/部分转入券商
  3) 券商端也要有同样分区，避免把"可投资资金"和"暂存资金"混在一起

**设计定论（必须明确）**：
- **父账户（ROOT）** = 平台容器/分组节点（余利宝、稳利宝、华宝证券等），用于UI聚合与筛选
- **子账户（BUCKET/叶子账户）** = 资金信封（生活费/房租/项目/理财金/安全金/待分配等），真实金额只存在子账户
- **父账户余额表述（关键）**：
  - 父账户不参与任何记账分录，禁止余额编辑，balance字段不作为真实余额来源（保持0或不可编辑）
  - 父账户展示余额 = Σ(子账户叶子余额)（仅展示层聚合）
- **ledger_posting.account_id** 只允许引用【叶子账户】（即不存在任何child的账户）。父账户禁止出现在任何记账分录中（避免把"分组节点"当资金池导致双算/不可追溯）

**fund_usage（资金用途）说明**：
- `SPENDABLE`（可支出）：允许日常支出/生活消费
- `RESERVED`（专款）：房租/项目/安全金等，禁止日常支出和默认禁止投资
- `INVESTABLE`（可投资）：可用于投资如ETF/逆回购等，默认不用于日常支出
- **约束范围**：只对`account_kind=REAL`且`account_type=CASH`且为叶子账户的场景做约束校验。对VIRTUAL/POSITION/LIABILITY等账户，fund_usage仅保留默认值但不参与业务校验，避免错误阻断

**待分配子账户（必须）**：
- 每个"平台父账户"必须至少有一个默认子账户：xxx_unallocated（待分配/自由资金）
- 工资/转入平台的资金默认进入待分配，再由用户分配到生活费/房租/项目/理财金等
- 这样能覆盖"非交易日暂存""还没来得及分配"的真实场景，不会误花专款

**叶子账户判定规则**：
- 叶子账户=不存在任何child的账户；只有叶子账户允许出现在ledger_posting.account_id

**父账户UI行为**：
- 父账户只展示汇总，不允许点进去做"支出/投资/转账来源"选择

**防呆规则（后端必须校验）**：
1. **EXPENSE（消费支出）**：只能从 `fund_usage=SPENDABLE` 的现金叶子账户扣款；否则拒绝并提示"这是专款/理财金，不允许消费"
2. **投资类 BUY/SUBSCRIPTION（ETF/基金/股票等）**：默认只能从 `fund_usage=INVESTABLE` 的账户作为资金来源；如要允许生活费临时投资必须显式确认（可选增强）
3. **任何扣减**：都要检查 `balance - reserved_amount >= 需要金额`（避免负数）

**交易/记账流转**：
1. **分配（ALLOCATE）**：把父账户自由资金分配到子账户
   - 以 TRANSFER_OUT/TRANSFER_IN 成对模型实现（保持现有transfer模型）：
     - OUT：父账户的待分配子账户 -> 目标子账户（同平台内部）
     - IN：目标子账户入账
   - UI用 biz_group_key 合并展示为"余利宝资金分配"
2. **跨平台迁移（MIGRATE）**：余利宝的多个分区在交易日转入华宝证券
   - 允许一次物理转账在系统里拆成多个"分区转移对"，全部共享同一个 biz_group_key
   - 例：余利宝房租/项目/理财金 -> 华宝对应分区（多对转账，UI合并为1条"余利宝->华宝"并显示分区明细）

**示例账户树（初始化数据）**：
```
- 余利宝(父)
  - 余利宝_待分配(unallocated, SPENDABLE)
  - 余利宝_生活费(SPENDABLE)
  - 余利宝_房租(RESERVED)
  - 余利宝_项目(RESERVED)
  - 余利宝_理财金(INVESTABLE)
- 稳利宝(父)
  - 稳利宝_待分配(unallocated)
  - 稳利宝_房租(RESERVED)
  - 稳利宝_项目(RESERVED)
  - 稳利宝_安全金(RESERVED)
- 华宝证券(父)
  - 华宝_待分配(unallocated)
  - 华宝_房租(RESERVED)
  - 华宝_项目(RESERVED)
  - 华宝_理财金(INVESTABLE)
```

**端到端流程示例**：
1. 工资到账：银行卡 -> 余利宝_待分配（TRANSFER_IN）
2. 资金分配：余利宝_待分配 -> 余利宝_生活费/房租/项目/理财金（TRANSFER_OUT/TRANSFER_IN成对）
3. 非交易日暂存：资金在余利宝各子账户中
4. 交易日迁移：余利宝_房租/项目/理财金 -> 华宝对应子账户（跨平台转账，多对转账共享biz_group_key）
5. 消费：只能从余利宝_生活费（fund_usage=SPENDABLE）扣款
6. ETF投资：只能从华宝_理财金（fund_usage=INVESTABLE）扣款

```sql
--    - 生成 ledger_txn (txn_type='EXPENSE')
--    - 生成 ledger_posting: CREDIT LIABILITY (负债增加), DEBIT EXPENSE (费用增加)
--    - 创建 debt_contract 和 debt_installment 记录
-- 2. 每期还款时：
--    - 生成 ledger_txn (txn_type='EXPENSE' 或 'TRANSFER_OUT')
--    - 生成 ledger_posting: DEBIT LIABILITY (负债减少), CREDIT CASH (现金减少)
--    - 更新 debt_installment.status='PAID', paid_amount, paid_date
-- 3. 手续费处理：
--    - 可计入费用账户（FEE）或利息支出（EXPENSE）
--    - 或作为负债的一部分（计入fee_total）
```

#### 4.3.2 ledger_posting - 分录表
```sql
CREATE TABLE `ledger_posting` (
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '分录ID',
  `txn_id` VARCHAR(64) NOT NULL COMMENT '交易事件ID（外键ledger_txn.txn_id）',
  `posting_type` ENUM('DEBIT', 'CREDIT') NOT NULL COMMENT '借贷方向（DEBIT=借方，对于资产类账户表示增加，对于负债类账户表示减少；CREDIT=贷方，对于资产类账户表示减少，对于负债类账户表示增加；amount永远为正数，方向由posting_type决定）',
  `account_id` BIGINT NOT NULL COMMENT '账户ID（外键accounts.id）',
  `account_type` ENUM('CASH', 'POSITION', 'FEE', 'INCOME', 'EXPENSE', 'LIABILITY', 'RECEIVABLE') NOT NULL COMMENT '账户类型（CASH=现金，现金账户；POSITION=持仓，持有的基金/股票；FEE=费用，手续费；INCOME=收入，投资收益；EXPENSE=支出，日常消费；LIABILITY=负债，欠款；RECEIVABLE=应收，应该收到但未到账的钱）',
  `amount` DECIMAL(18, 2) NOT NULL COMMENT '金额（永远为正数，方向由posting_type决定）',
  `shares` DECIMAL(20, 6) NULL COMMENT '份额（份额即持有的基金/ETF数量，类似股票股数，持仓类分录，永远为正数，方向由posting_type决定）',
  `currency` ENUM('CNY', 'USD', 'HKD') NOT NULL DEFAULT 'CNY' COMMENT '货币',
  `note` VARCHAR(255) NULL COMMENT '分录备注',
  `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_txn_id` (`txn_id`),
  KEY `idx_account_id` (`account_id`),
  KEY `idx_posting_type` (`posting_type`),
  KEY `idx_account_type` (`account_type`),
  CONSTRAINT `chk_amount_positive` CHECK (`amount` > 0),
  CONSTRAINT `chk_shares_non_negative` CHECK (`shares` IS NULL OR `shares` >= 0)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_general_ci COMMENT='分录表';
```

**借贷方向规则（全局统一，必须严格遵守）**：

采用方案A：`amount` 永远为正数，`posting_type` 决定方向

**资产类账户（CASH/POSITION/RECEIVABLE）**：
- `DEBIT` = 增加资产（余额增加）
- `CREDIT` = 减少资产（余额减少）
- **专有名词公式**：`balance = initial_balance + SUM(DEBIT金额) - SUM(CREDIT金额)`
- **解释公式**：余额 = 初始余额 + 所有增加金额 - 所有减少金额
- **通俗解释**：
  - `initial_balance`：账户创建时的初始金额
  - `SUM(DEBIT金额)`：所有增加账户余额的交易金额总和（如收入、转入）
  - `SUM(CREDIT金额)`：所有减少账户余额的交易金额总和（如支出、转出）
  - 为什么是"DEBIT减CREDIT"：因为对于资产类账户，借方表示增加，贷方表示减少

**负债类账户（LIABILITY）**：
- `DEBIT` = 减少负债（欠款减少）
- `CREDIT` = 增加负债（欠款增加）
- **专有名词公式**：`balance = initial_balance + SUM(CREDIT金额) - SUM(DEBIT金额)`
- **解释公式**：负债余额 = 初始负债 + 所有增加负债的金额总和 - 所有减少负债的金额总和
- **通俗解释**：
  - 为什么是"CREDIT减DEBIT"：因为对于负债类账户，贷方表示增加负债（欠款增加），借方表示减少负债（还款）
  - 余额为正数表示欠款金额

**费用类账户（FEE/EXPENSE）**：
- `DEBIT` = 费用发生（费用增加）
- `CREDIT` = 费用冲减（费用减少，少见）
- **专有名词公式**：`balance = SUM(DEBIT金额) - SUM(CREDIT金额)`
- **解释公式**：费用总额 = 所有发生的费用金额总和 - 所有冲减的费用金额总和
- **通俗解释**：
  - 为什么没有初始余额：费用账户通常从0开始
  - 为什么是"DEBIT减CREDIT"：借方表示费用发生，贷方表示费用冲减（少见）
  - 费用余额为正数表示费用总额

**收入类账户（INCOME）**：
- `DEBIT` = 收入冲减（收入减少，少见）
- `CREDIT` = 收入发生（收入增加）
- **专有名词公式**：`balance = SUM(CREDIT金额) - SUM(DEBIT金额)`
- **解释公式**：收入总额 = 所有发生的收入金额总和 - 所有冲减的收入金额总和
- **通俗解释**：
  - 为什么是"CREDIT减DEBIT"：贷方表示收入发生，借方表示收入冲减（少见）
  - 收入余额为正数表示收入总额

**关键设计说明**：
- `ledger_posting` 是唯一真相源，所有余额/持仓/费用/收入均由分录计算得出
- `ledger_txn` 仅作为业务事件头，不存储金额/份额（或仅作为派生缓存，禁止手工编辑）
- **重要约束**：`ledger_posting.account_id` 只允许引用【叶子账户】（即不存在任何child的账户）。父账户禁止出现在任何记账分录中（避免把"分组节点"当资金池导致双算/不可追溯）
- UI合并展示：按 `biz_group_key` 或 `order_id` 分组，用户看到的是业务事件，不是分录
- 持仓计算：按 `product_id` 汇总 `ledger_posting` 中 `account_type='POSITION'` 的 `shares`（DEBIT增加，CREDIT减少，shares永远为正数）

#### 4.3.3 统一记账入口（一个入口覆盖所有业务类型）

**设计目标**：前端只保留一个"记账/记一笔"入口，覆盖所有业务类型，简化用户操作，统一数据结构。

**支持的业务类型**：

1. **日常类**
   - `EXPENSE`：支出（消费）
   - `INCOME`：收入
   - `TRANSFER_OUT`/`TRANSFER_IN`：转账（成对出现）

2. **资金类**（必须关联原交易）
   - `REIMBURSE_IN`：退款（必须设置`related_txn_id`，指向原交易）
   - `REIMBURSE_OUT`：报销（必须设置`related_txn_id`，指向原支出）

3. **理财交易类**
   - `BUY`/`SELL`：场内买入/卖出（ETF/股票等）
   - `SUBSCRIPTION`/`REDEMPTION`：场外申购/赎回
   - `CUSTODY_TRANSFER`：转托管（持仓从A账户迁到B账户，无现金流，只有持仓/成本迁移）
   - `BOND_REPO`：国债逆回购（现金增强，不生成持仓，见4.3.4节）

**统一记账入口的UI交互流程**：

1. **第一步：选择业务类型**
   - 用户点击"记一笔"按钮，弹出统一记账中心弹窗/抽屉
   - 选择业务类型：支出/收入/转账/退款/报销/买入/卖出/申购/赎回/转托管/逆回购

2. **第二步：根据类型展示不同字段**
   - **支出/收入**：账户选择、金额、备注、日期（支持组合支付：多个资金来源）
   - **转账**：转出账户、转入账户、金额、备注、日期
   - **退款/报销**：原交易选择（从历史交易列表选择）、退款/报销金额、到账账户、备注、日期
   - **买入/卖出**：产品选择、账户选择、价格/净值、份额/金额、手续费、成交日期（支持组合支付）
   - **申购/赎回**：产品选择、账户选择、金额/份额、预期确认日期、备注（支持组合支付）
   - **转托管**：产品选择、转出账户、转入账户、份额、成本、转出日期
   - **逆回购**：产品选择（asset_type=BOND_REPO）、账户选择、本金、到期日期、备注

3. **第三步：生成统一数据结构**
   - 所有业务类型最终都生成`ledger_txn`和`ledger_posting`记录
   - 退款/报销自动设置`related_txn_id`和`relation_type`
   - 组合支付自动生成多条`ledger_posting`记录（或`order_funding_line`记录）

**关键业务规则**：
- 退款/报销必须关联原交易，从原交易详情页可以查看：已退款金额、已报销金额、剩余可退/可报金额
- 组合支付：同一笔交易/订单可由多个账户共同支付（现金+白条/负债等）
- 转托管：无现金流，只迁移持仓和成本，生成`CUSTODY_TRANSFER`类型的交易
- 逆回购：不生成持仓，只占用资金和生成利息（见4.3.4节）

#### 4.3.4 交易关联关系模型

**设计目标**：支持交易之间的关联关系，实现退款/报销追溯、转账成对展示、撤销关联等功能。

**关联字段设计**（在`ledger_txn`表中）：
- `related_txn_id` VARCHAR(64) NULL：关联的原交易txn_id（用于退款/报销/撤销/转托管等场景）
- `related_order_id` VARCHAR(64) NULL：关联的原订单号（用于订单级退款/撤单）
- `relation_type` ENUM('NONE','TRANSFER_PAIR','REFUND','REFUND_OF','REIMBURSE','REIMBURSEMENT_OF','REVERSAL','CUSTODY_TRANSFER_OF')：关联类型

**关联优先规则**：
- 关联优先使用`related_txn_id`/`related_order_id`进行语义关联
- `biz_group_key`用于UI聚合展示，不替代语义关联（例如转账成对使用`biz_group_key`，但退款/报销必须使用`related_txn_id`）

**关联场景定义**：

1. **转账成对（TRANSFER_PAIR）**
   - 使用`biz_group_key`关联：TRANSFER_OUT和TRANSFER_IN共享同一个`biz_group_key`
   - `relation_type`保持为`NONE`（不设置`related_txn_id`，因为使用`biz_group_key`）
   - UI展示：按`biz_group_key`合并为一条"转账"记录
   - 查询示例：`SELECT * FROM ledger_txn WHERE biz_group_key = 'TRF-20240115-001'`

2. **退款（REFUND / REFUND_OF）**
   - 必须设置`related_txn_id`指向原交易（原交易可以是EXPENSE或订单生成的交易）
   - `relation_type = 'REFUND_OF'`（推荐，语义更清晰）或`'REFUND'`（兼容旧值）
   - 支持多次部分退款：允许多笔退款交易指向同一原交易
   - 退款金额可以小于原交易金额
   - 查询原交易的退款总额：
     ```sql
     SELECT 
       t.txn_id,
       t.txn_type,
       SUM(IFNULL(p.amount, 0)) as original_amount,
       COALESCE(SUM(CASE WHEN r.relation_type = 'REFUND' THEN rp.amount ELSE 0 END), 0) as refunded_total,
       SUM(IFNULL(p.amount, 0)) - COALESCE(SUM(CASE WHEN r.relation_type = 'REFUND' THEN rp.amount ELSE 0 END), 0) as remaining
     FROM ledger_txn t
     LEFT JOIN ledger_posting p ON t.txn_id = p.txn_id AND p.posting_type = 'DEBIT' AND p.account_type = 'EXPENSE'
     LEFT JOIN ledger_txn r ON r.related_txn_id = t.txn_id AND r.relation_type = 'REFUND'
     LEFT JOIN ledger_posting rp ON r.txn_id = rp.txn_id AND rp.posting_type = 'DEBIT' AND rp.account_type = 'CASH'
     WHERE t.txn_id = ?
     GROUP BY t.txn_id
     ```

3. **报销（REIMBURSE / REIMBURSEMENT_OF）**
   - 必须设置`related_txn_id`指向原交易（原交易必须是EXPENSE且标记为可报销）
   - `relation_type = 'REIMBURSEMENT_OF'`（推荐，语义更清晰）或`'REIMBURSE'`（兼容旧值）
   - 支持多次部分报销：允许多笔报销交易指向同一原交易
   - 报销金额可以小于原交易金额
   - 查询原交易的报销总额（类似退款查询）：
     ```sql
     SELECT 
       t.txn_id,
       COALESCE(SUM(CASE WHEN r.relation_type = 'REIMBURSE' THEN rp.amount ELSE 0 END), 0) as reimbursed_total
     FROM ledger_txn t
     LEFT JOIN ledger_txn r ON r.related_txn_id = t.txn_id AND r.relation_type = 'REIMBURSE'
     LEFT JOIN ledger_posting rp ON r.txn_id = rp.txn_id AND rp.posting_type = 'DEBIT' AND rp.account_type = 'CASH'
     WHERE t.txn_id = ?
     GROUP BY t.txn_id
     ```

4. **撤销（REVERSAL）**
   - 必须设置`related_txn_id`指向被撤销的交易
   - `relation_type = 'REVERSAL'`
   - 撤销交易生成反向分录，不删除原记录
   - 查询被撤销的交易：`SELECT * FROM ledger_txn WHERE related_txn_id = ? AND relation_type = 'REVERSAL'`

5. **转托管（CUSTODY_TRANSFER_OF）**
   - 转托管交易（`txn_type='CUSTODY_TRANSFER'`）可以设置`related_txn_id`指向发起事件（可选）
   - `relation_type = 'CUSTODY_TRANSFER_OF'`（可选，用于关联同一批次的多笔转托管）
   - 转托管无现金流，只迁移持仓和成本：
     - 转出账户：POSITION CREDIT（份额和成本减少）
     - 转入账户：POSITION DEBIT（份额和成本增加）
   - 查询转托管关联：`SELECT * FROM ledger_txn WHERE related_txn_id = ? AND relation_type = 'CUSTODY_TRANSFER_OF'`

**应用层约束**（MySQL CHECK约束不支持引用外键列，必须在应用层校验）：
1. `related_txn_id`不能等于`txn_id`（防止自引用）
2. `relation_type='REFUND'`、`'REFUND_OF'`、`'REIMBURSE'`或`'REIMBURSEMENT_OF'`时，`related_txn_id`必须非空
3. `relation_type='REVERSAL'`时，`related_txn_id`必须非空且指向有效交易
4. 退款/报销总额不能超过原交易金额（应用层校验，允许部分退款/报销）
5. `relation_type='CUSTODY_TRANSFER_OF'`时，`txn_type`必须为`'CUSTODY_TRANSFER'`

**UI/查询口径**：
- 原交易详情页显示：
  - `refunded_total`：已退款总额（从关联的REFUND交易汇总）
  - `reimbursed_total`：已报销总额（从关联的REIMBURSE交易汇总）
  - `remaining`：剩余可退/可报金额 = 原交易金额 - refunded_total - reimbursed_total
  - 退款/报销明细列表：查询所有`related_txn_id=原交易txn_id`且`relation_type='REFUND'/'REIMBURSE'`的交易

**统计口径说明**：
- 退款/报销金额从关联交易的`ledger_posting`中汇总（`posting_type='DEBIT'`且`account_type='CASH'`）
- 不允许重复计入：原交易的EXPENSE分录和退款/报销的CASH分录分别计入不同账户，不会重复统计
- 原交易详情页的`refunded_total`/`reimbursed_total`仅用于展示，不影响余额计算（余额由分录自动计算）

**最小验算用例**：
1. 现金账户初始10000，下单买入5000：
   - 分录：CREDIT CASH 5000（现金减少）
   - 余额：10000 - 5000 = 5000 ✓
2. 转账1000：A账户减少，B账户增加：
   - 分录：CREDIT A账户 1000，DEBIT B账户 1000
   - 合计不变 ✓
3. 信用卡消费1000（负债增加）：
   - 分录：CREDIT LIABILITY 1000（负债增加），DEBIT EXPENSE 1000（费用增加）
   - 负债余额：0 + 1000 = 1000（欠款1000）✓

- **到期处理**：确认订单（CONFIRMED），必须按顺序：1) 校验订单到期；2) 释放占用（accounts.reserved_amount -= principal）；3) 生成利息ledger_txn（CASH DEBIT(interest) + INCOME CREDIT(interest)，记到同一个发起子账户）
- **看板计算**：repo_locked_amount必须用订单汇总计算（Σ(所有REPO订单的principal，status=LOCKED/PENDING/ACTIVE)），不能用reserved_amount的一部分

#### 4.4.4 组合支付（Split Payment / Multi-Funding）

**设计目标**：支持一笔消费或订单由多个账户共同支付，例如：现金3000 + 白条(负债)2000。

**消费组合支付示例**：

场景：用户消费5000元，使用现金3000元 + 白条2000元

**分录模板**：
```
ledger_txn:
  txn_id: "TXN-20240115-EXP-001"
  txn_type: "EXPENSE"
  note: "消费5000元（现金3000+白条2000）"

ledger_posting:
  # 费用增加（总额）
  DEBIT EXPENSE 5000.00
  
  # 现金减少（部分）
  CREDIT CASH 3000.00 (account_id=现金账户)
  
  # 负债增加（部分）
  CREDIT LIABILITY 2000.00 (account_id=白条账户)
```

**订单组合支付示例**：

场景：用户买入ETF 5000元，使用生活费账户3000元 + 理财金账户2000元

**步骤1：创建订单（PENDING状态）**
```
orders:
  order_id: "ORD-20240115-001"
  amount: 5000.00
  status: "PENDING"

order_funding_line:
  line_no=1: account_id=生活费账户, amount=3000.00
  line_no=2: account_id=理财金账户, amount=2000.00

accounts:
  生活费账户.reserved_amount += 3000.00
  理财金账户.reserved_amount += 2000.00
```

**步骤2：确认结算（CONFIRMED状态）**
```
settlement_confirm:
  order_id: "ORD-20240115-001"
  confirm_amount: 5000.00
  confirm_shares: 4854.37

ledger_txn:
  txn_id: "TXN-20240115-BUY-001"
  txn_type: "BUY"
  order_id: "ORD-20240115-001"

ledger_posting:
  # 持仓增加
  DEBIT POSITION shares=4854.37, amount=5000.00
  
  # 现金减少（按funding_line拆分）
  CREDIT CASH 3000.00 (account_id=生活费账户)
  CREDIT CASH 2000.00 (account_id=理财金账户)

accounts:
  生活费账户.reserved_amount -= 3000.00
  生活费账户.balance -= 3000.00
  理财金账户.reserved_amount -= 2000.00
  理财金账户.balance -= 2000.00
```

**关键业务规则**：
1. 创建订单时：写入`order_funding_line`，分别增加各`account.reserved_amount`
2. 取消订单时：逐条释放各`account.reserved_amount`，删除`order_funding_line`记录
3. 确认结算时：按`order_funding_line`生成多条CASH CREDIT分录，并清零对应`account.reserved_amount`
4. 组合支付总额必须等于订单金额：`Σ(order_funding_line.amount) = orders.amount`（应用层校验）
5. 每个资金来源账户的可用余额必须足够：`account.balance - account.reserved_amount >= funding_line.amount`（应用层校验）

**白条/负债支付说明**：
- 白条支付时，负债账户（LIABILITY）的CREDIT表示负债增加（欠款增加）
- 分录：`CREDIT LIABILITY amount`（负债增加）+ `DEBIT EXPENSE amount`（费用增加）
- 后续还款时：`DEBIT LIABILITY amount`（负债减少）+ `CREDIT CASH amount`（现金减少）

**快照脏数据标记与重算策略**：

1. **脏数据标记规则**：
   - 当任意历史流水/订单确认被修改（金额、份额、日期、费用等）时：
     - 标记该日期及之后所有快照为 `is_dirty=1`
     - 设置 `dirty_from_date` = 被修改的最早日期
   
2. **重算策略**：
   - 触发机制：历史修正/冲正 -> 标脏 -> 触发 `SNAPSHOT_RECALC`（可用 `py_job` 跟踪）
   - 后台定时任务：从 `dirty_from_date` 起，使用 `(T-1 快照 + T 日流水)` 逐日链式重建，直到最新
   - 性能保护：
     - >1 年前的历史修正：建议用"今日 ADJUST 分录"平账，或只允许低峰重算
     - 重算时加锁，避免并发冲突
   - 前端提示：重算进行中时，前端显示"历史修正正在回算中，请稍候"
   - 异步执行：重算任务通过 `py_job` 表跟踪，前端可轮询进度

3. **成本核算方法**：
   - 默认：`AVERAGE`（加权平均成本）
   - 可选：`FIFO`（先进先出，需启用 `position_lots` 表，Phase 2 增强）

4. **收益拆分**：
   - `realized_pnl`：从卖出/赎回的流水计算（卖出价格 - 成本）
   - `unrealized_pnl`：从持仓市值 - 成本计算
   - `income_pnl`：从分红/利息的流水汇总

## 5. 订单/结算设计详解

### 5.1 场外T+N结算流程

#### 5.1.1 买入/申购流程（关键：下单阶段是"占用/冻结"，不是"已完成扣款"）

**步骤1：用户发起买入/申购（下单阶段）**
```
1. 创建 orders 记录
   - order_id: "ORD-20240115-001"
   - status: "PENDING"
   - amount: 5000.00
   - requested_at: "2024-01-15 14:30:00"
   - trade_date: "2024-01-15" (14:30 < 15:00 cutoff)
   - expected_nav_date: "2024-01-15"
   - expected_confirm_date: "2024-01-16" (T+1)

2. 资金占用（关键！）
   - accounts.reserved_amount += 5000.00（占用增加，不直接改balance）
   - 不生成 ledger_txn 和 ledger_posting（此时还未真正扣款）

3. 可用购买力计算
   - available_for_trade = balance - reserved_amount
   - 例如：balance=10000, reserved=5000 → available=5000
```

**步骤2：待结算阶段**
```
- 首页显示"待结算清单"（orders WHERE status='PENDING'）
- 显示预期确认日期、预期净值日期、预估份额
- 显示占用金额（reserved_amount）
```

**步骤3：手工确认结算（确认阶段）**
```
1. 用户核对实际到账情况
   - 创建 settlement_confirm 记录
   - confirm_date: "2024-01-16"（可人工覆盖）
   - nav_date: "2024-01-15"（可人工覆盖）
   - confirm_nav: 1.0305（可人工覆盖）
   - confirm_shares: 4851.55（可人工覆盖）

2. 释放占用并生成真实流水
   - accounts.reserved_amount -= 5000.00（释放占用）
   - 生成 ledger_txn（txn_type='SUBSCRIPTION', status='CONFIRMED'）
   - 生成 ledger_posting（CREDIT CASH 5004.25，现金减少，实际扣款总额 = 5000 + 4.25）
   - 生成 ledger_posting（DEBIT POSITION shares=4851.55, amount=5000.00，持仓增加，成本口径不含手续费）
   - 生成 ledger_posting（DEBIT FEE amount=4.25，手续费）

3. 更新订单状态
   - orders.status = 'CONFIRMED'

4. 如有差异，生成调整分录
   - 如果预估份额 ≠ 实际份额，生成调整分录
```

**三口径公式**：
- **专有名词公式**：`balance = initial_balance + SUM(DEBIT) - SUM(CREDIT)`
  - **解释公式**：账面余额 = 初始余额 + 所有增加金额 - 所有减少金额
  - **通俗解释**：初始余额加上所有借方金额，减去所有贷方金额
- **专有名词公式**：`available_for_trade = balance - reserved_amount`
  - **解释公式**：可用购买力 = 账面余额 - 占用金额
  - **通俗解释**：账户中的总金额减去已下单但未确认时占用的金额（冻结资金），因为这部分钱已经"预定"给某个订单了，不能再用于其他购买
- **专有名词公式**：`available_for_withdraw = balance - 未交收卖出款 - reserved_amount`
  - **解释公式**：可取资金 = 账面余额 - 未交收卖出款 - 占用金额
  - **通俗解释**：账面余额减去卖出后还未到账的钱（未交收卖出款），再减去占用金额，最终得到真正可以取出来用的钱

**占用/冻结与可用资金口径（必须明确）**：
- `accounts.reserved_amount` 的含义：
  - 下单阶段：`reserved_amount += 订单金额`（占用增加，不生成流水，不扣款）
  - 结算确认：`reserved_amount -= 订单金额`（释放占用，生成真实流水）
  - 取消订单：`reserved_amount -= 订单金额`（释放占用，不生成流水）
- `available_for_trade`（可用购买力）：
  - **专有名词公式**：`available_for_trade = balance - reserved_amount`
  - **解释公式**：可用购买力 = 账面余额 - 占用金额
  - **通俗解释**：可用于下单的资金（已扣除占用），因为占用金额是已下单但未确认时占用的金额，这部分钱已经"预定"给某个订单了
- `available_for_withdraw`（可取资金）：
  - **专有名词公式**：`available_for_withdraw = balance - 未交收卖出款 - reserved_amount`
  - **解释公式**：可取资金 = 账面余额 - 未交收卖出款 - 占用金额
  - **未交收卖出款计算公式**：`未交收卖出款 = SUM(赎回订单金额 WHERE status='PENDING' AND expected_confirm_date > CURDATE())`
  - **通俗解释**：
    - 未交收卖出款：卖出基金后，钱不会立即到账（T+N），在到账前不能取用
    - 为什么减去未交收卖出款：这部分钱虽然显示在余额里，但实际还没到账，不能取
    - 为什么减去占用金额：这部分钱已经预定给某个订单，不能取
    - 最终结果：真正可以取出来用的钱
  - UI必须能解释"这笔钱为何不能提现"，展示预计可取日


### 5.4 关键口径与计算公式汇总

#### 5.4.1 账户余额计算公式

**资产类账户（CASH/POSITION/RECEIVABLE）**：
- **专有名词公式**：`balance = initial_balance + SUM(DEBIT金额) - SUM(CREDIT金额)`
- **解释公式**：余额 = 初始余额 + 所有增加金额 - 所有减少金额
- **SQL实现**：
```sql
balance = initial_balance + SUM(
  CASE 
    WHEN posting_type = 'DEBIT' THEN amount
    WHEN posting_type = 'CREDIT' THEN -amount
  END
)
```
- **通俗解释**：
  - `initial_balance`：账户创建时的初始金额
  - `SUM(DEBIT金额)`：所有增加账户余额的交易金额总和（如收入、转入）
  - `SUM(CREDIT金额)`：所有减少账户余额的交易金额总和（如支出、转出）
  - 为什么是"DEBIT减CREDIT"：因为对于资产类账户，借方表示增加，贷方表示减少

**负债类账户（LIABILITY）**：
- **专有名词公式**：`balance = initial_balance + SUM(CREDIT金额) - SUM(DEBIT金额)`
- **解释公式**：负债余额 = 初始负债 + 所有增加负债的金额总和 - 所有减少负债的金额总和
- **SQL实现**：
```sql
balance = initial_balance + SUM(
  CASE 
    WHEN posting_type = 'CREDIT' THEN amount  -- 负债增加
    WHEN posting_type = 'DEBIT' THEN -amount  -- 负债减少
  END
)
```
- **通俗解释**：
  - 为什么是"CREDIT减DEBIT"：因为对于负债类账户，贷方表示增加负债（欠款增加），借方表示减少负债（还款）
  - 余额为正数表示欠款金额

**费用类账户（FEE/EXPENSE）**：
- **专有名词公式**：`balance = SUM(DEBIT金额) - SUM(CREDIT金额)`
- **解释公式**：费用总额 = 所有发生的费用金额总和 - 所有冲减的费用金额总和
- **SQL实现**：
```sql
balance = SUM(
  CASE 
    WHEN posting_type = 'DEBIT' THEN amount
    WHEN posting_type = 'CREDIT' THEN -amount
  END
)
```
- **通俗解释**：
  - 为什么没有初始余额：费用账户通常从0开始
  - 为什么是"DEBIT减CREDIT"：借方表示费用发生，贷方表示费用冲减（少见）
  - 费用余额为正数表示费用总额

**收入类账户（INCOME）**：
- **专有名词公式**：`balance = SUM(CREDIT金额) - SUM(DEBIT金额)`
- **解释公式**：收入总额 = 所有发生的收入金额总和 - 所有冲减的收入金额总和
- **SQL实现**：
```sql
balance = SUM(
  CASE 
    WHEN posting_type = 'CREDIT' THEN amount
    WHEN posting_type = 'DEBIT' THEN -amount
  END
)
```
- **通俗解释**：
  - 为什么是"CREDIT减DEBIT"：贷方表示收入发生，借方表示收入冲减（少见）
  - 收入余额为正数表示收入总额

#### 5.4.2 资金口径计算公式

**可用购买力（Available for Trade）**：
- **专有名词公式**：`available_for_trade = balance - reserved_amount`
- **解释公式**：可用购买力 = 账面余额 - 占用金额
- **SQL实现**：
```sql
available_for_trade = balance - reserved_amount
```
- **通俗解释**：
  - `balance`：账户中的总金额
  - `reserved_amount`：已下单但未确认时占用的金额（冻结资金）
  - 为什么减去占用金额：因为这部分钱已经"预定"给某个订单了，不能再用于其他购买

**未交收卖出款（Unsettled Sale Proceeds）**：
- **专有名词公式**：`unsettled_sale_proceeds = SUM(待确认卖出订单的预期到账金额)`
- **解释公式**：未交收卖出款 = 所有已卖出但还未到账的金额总和
- **SQL实现**：
```sql
unsettled_sale_proceeds = SUM(
  CASE 
    WHEN o.order_type IN ('SELL', 'REDEMPTION') 
      AND o.status = 'PENDING' 
      AND o.expected_confirm_date > CURDATE()
    THEN COALESCE(sc.confirm_amount, o.shares * COALESCE(n.nav, 0))
    ELSE 0
  END
)
FROM orders o
LEFT JOIN settlement_confirm sc ON o.order_id = sc.order_id
LEFT JOIN nav n ON o.product_id = n.product_id AND n.nav_date = o.expected_nav_date
WHERE o.user_id = ?
```
- **通俗解释**：
  - 为什么需要这个：卖出基金后，钱不会立即到账（T+N），在到账前不能取用
  - 如何计算：查找所有状态为"待确认"的卖出订单，计算它们的预期到账金额
  - 查找条件：订单类型为卖出或赎回，状态为待确认，预期确认日期大于今天

**可取资金（Available for Withdraw）**：
- **专有名词公式**：`available_for_withdraw = balance - unsettled_sale_proceeds - reserved_amount`
- **解释公式**：可取资金 = 账面余额 - 未交收卖出款 - 占用金额
- **SQL实现**：
```sql
available_for_withdraw = balance - unsettled_sale_proceeds - reserved_amount
```
- **通俗解释**：
  - 为什么减去未交收卖出款：这部分钱虽然显示在余额里，但实际还没到账，不能取
  - 为什么减去占用金额：这部分钱已经预定给某个订单，不能取
  - 最终结果：真正可以取出来用的钱

#### 5.4.3 持仓计算

**持仓份额**：
- **专有名词公式**：`shares = SUM(CASE WHEN posting_type='DEBIT' THEN shares ELSE -shares END)`
- **解释公式**：持仓份额 = 所有买入的份额总和 - 所有卖出的份额总和
- **SQL实现**：
```sql
shares = SUM(
  CASE 
    WHEN posting_type = 'DEBIT' AND account_type = 'POSITION' THEN shares
    WHEN posting_type = 'CREDIT' AND account_type = 'POSITION' THEN -shares  -- shares永远为正数，方向由posting_type决定
    ELSE 0
  END
)
FROM ledger_posting
WHERE account_type = 'POSITION' AND product_id = ?
```
- **通俗解释**：
  - 为什么DEBIT是加，CREDIT是减：DEBIT表示持仓增加（买入），CREDIT表示持仓减少（卖出）
  - shares永远为正数：买入1000份，卖出300份，剩余700份

**持仓成本（加权平均）**：
- **专有名词公式**：`cost = SUM(amount) / SUM(shares)`（所有买入交易的amount总和 / 所有买入交易的shares总和）
- **解释公式**：持仓成本 = 所有买入的总金额 / 所有买入的总份额
- **SQL实现**：
```sql
cost = SUM(
  CASE 
    WHEN posting_type = 'DEBIT' AND account_type = 'POSITION' THEN amount
    WHEN posting_type = 'CREDIT' AND account_type = 'POSITION' THEN -amount  -- amount永远为正数，方向由posting_type决定
    ELSE 0
  END
) / NULLIF(shares, 0)
```
- **通俗解释**：
  - 为什么这样算：假设分3次买入，每次价格不同，加权平均就是总成本除以总份额
  - 例子：第1次1.2元买1000份（1200元），第2次1.3元买1000份（1300元），成本 = (1200+1300)/(1000+1000) = 1.25元/份

#### 5.4.4 收益拆分

**已实现收益（Realized PnL）**：
- **专有名词公式**：`realized_pnl = (卖出净到账金额) - (卖出部分成本) - (手续费)`
- **解释公式**：已实现盈亏 = 卖出后实际收到的钱 - 卖出那部分的本金 - 手续费
- **SQL实现**：
```sql
-- 由postings推导：现金净流入 - 手续费 - POSITION成本扣减
realized_pnl = (
  SELECT COALESCE(SUM(CASE WHEN p.account_type = 'CASH' AND p.posting_type = 'DEBIT' THEN p.amount ELSE 0 END), 0)  -- 现金净流入
  - COALESCE(SUM(CASE WHEN p.account_type = 'FEE' AND p.posting_type = 'DEBIT' THEN p.amount ELSE 0 END), 0)  -- 手续费
  - COALESCE(SUM(CASE WHEN p.account_type = 'POSITION' AND p.posting_type = 'CREDIT' THEN p.amount ELSE 0 END), 0)  -- 成本扣减
)
FROM ledger_txn t
INNER JOIN ledger_posting p ON t.txn_id = p.txn_id
WHERE t.txn_type IN ('SELL', 'REDEMPTION') AND t.user_id = ?
```
- **通俗解释**：
  - 卖出净到账金额：卖出后实际收到的钱（扣除手续费后）
  - 卖出部分成本：按成本核算方法（平均成本或FIFO）计算的卖出部分的本金
  - 手续费：卖出时的手续费
  - 为什么减去成本：因为成本是"本钱"，只有超过本钱的部分才是"赚的"
- **具体计算步骤（平均成本法）**：
  1. 计算卖出部分成本：`卖出成本 = 卖出份额 × 平均成本`
  2. 计算卖出净到账：`卖出净到账 = 卖出份额 × 卖出净值 - 手续费`
  3. 计算已实现盈亏：`已实现盈亏 = 卖出净到账 - 卖出成本`

**未实现收益（Unrealized PnL）**：
- **专有名词公式**：`unrealized_pnl = market_value - cost`
- **解释公式**：浮动盈亏 = 持仓市值 - 持仓成本
- **SQL实现**：
```sql
unrealized_pnl = (当前市值 - 持仓成本)
WHERE 当前市值 = 持仓份额 * 最新净值
```
- **通俗解释**：
  - 为什么这样算：当前值多少钱 - 花了多少钱 = 赚了或亏了多少
  - 正数表示盈利，负数表示亏损
  - 为什么叫"浮动"：因为还没卖出，价格会变动，盈亏也会变动
- **持仓市值计算公式**：
  - **专有名词公式**：`market_value = shares * nav`
  - **解释公式**：持仓市值 = 持仓份额 × 当前净值
  - **通俗解释**：
    - `shares`：持有的份额数量
    - `nav`：当前每份的价格（净值）
    - 为什么这样算：就像股票，股数 × 股价 = 总价值

**收入收益（Income PnL）**：
- **专有名词公式**：`income_pnl = SUM(分红金额 + 利息金额)`
- **解释公式**：收入收益 = 所有分红和利息的总和
- **SQL实现**：
```sql
income_pnl = SUM(
  CASE 
    WHEN txn_type IN ('DIVIDEND_CASH', 'DIVIDEND_REINVEST', 'INTEREST')
    THEN amount
    ELSE 0
  END
)
FROM ledger_posting
WHERE account_type = 'INCOME' AND user_id = ?
```
- **通俗解释**：
  - 分红：基金将收益分配给投资者的行为
  - 利息：存款或债券产生的收益
  - 收入收益是投资收益的一部分，与买卖盈亏不同

### 5.5 分红除权 vs 派息时间错配（两步分录，避免曲线断崖）

**问题**：分红除权日（EX_DATE）市值下跌，但派息日（PAY_DATE）才到账，如果只记录派息，会导致除权日净资产凭空暴跌。

**解决方案**：使用两步分录 + 虚拟应收账户

**步骤1：除权日（EX_DATE）**
```
1. 生成 ledger_txn（txn_type='DIVIDEND_EX_DATE'）
   - trade_date: EX_DATE
   - nav_date: EX_DATE
   - note: "分红除权"

2. 生成 ledger_posting（借：应收股利 RECEIVABLE）
   - DEBIT RECEIVABLE 100.00（应收股利增加）

3. 生成 ledger_posting（贷：投资收益 INCOME）
   - CREDIT INCOME 100.00（投资收益增加）

4. 持仓市值下跌（由行情自动更新）
   - 除权后净值 = 除权前净值 - 每股分红
   - 持仓市值相应下跌

5. 净资产不变（市值下跌 + 应收增加 = 0）
   - 净资产 = 持仓市值 + 现金 + 应收股利
   - 除权日：市值-100，应收+100，净资产不变 ✓
```

**步骤2：派息日（PAY_DATE）**
```
1. 生成 ledger_txn（txn_type='DIVIDEND_PAY_DATE'）
   - trade_date: PAY_DATE
   - confirm_date: PAY_DATE
   - note: "分红到账"

2. 生成 ledger_posting（借：现金 CASH）
   - DEBIT CASH 100.00（现金增加）

3. 生成 ledger_posting（贷：应收股利 RECEIVABLE）
   - CREDIT RECEIVABLE 100.00（应收减少）

4. 净资产不变（现金增加 + 应收减少 = 0）
   - 派息日：现金+100，应收-100，净资产不变 ✓
```

**关键点**：
- 除权日：市值下跌但应收增加，净资产不崩
- 派息日：现金增加但应收减少，净资产不崩
- 使用虚拟 RECEIVABLE 账户，避免曲线断崖

### 5.5 红利再投（DIVIDEND_REINVEST）

**流程**：
```
1. 除权日：同现金分红（生成应收）
2. 派息日：不生成现金分录，直接生成持仓增加分录
   - DEBIT POSITION（份额增加 = 分红金额 / 除权后净值）
   - CREDIT RECEIVABLE（应收减少）
```

```nginx
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        
        # WebSocket支持（如需要）
        proxy_http_version 1.1;

        proxy_set_header Connection "upgrade";
        
        # 超时设置

        proxy_read_timeout 60s;

    # 健康检查（可选）

    }
}
```

## 8. 回测实验室设计详解（一等模块）

### 8.1 回测实验室定位与原则

**定位**：回测实验室是系统的"一等模块"，用于策略参数优化与验证，与生产环境完全隔离。

**核心原则**：
1. **隔离原则**：回测只读行情与历史数据，不写生产账本真相表（ledger_posting/快照等）
2. **结果存储**：摘要指标存DB，明细数据存文件（csv.gz/parquet），节省存储
3. **参数推广**：最佳参数集可一键推广到生产配置（需二次确认）
4. **成本法一致**：回测与实盘使用相同的成本核算方法（AVERAGE/FIFO）

### 6.2 回测实验室最小闭环

#### 6.2.1 BacktestPlan（回测计划）

**职责**：定义实验配置，包括：
- 产品范围（product_ids JSON数组）
- 策略版本（strategy_key + strategy_version）
- 参数空间（param_space_json，定义参数范围/枚举值）
- 时间范围（start_date, end_date）
- 初始资金（initial_cash）
- 成本法（cost_method: AVERAGE/FIFO）
- 手续费模型（fee_model_json）

**示例配置**：
```json
{
  "product_ids": [1, 2, 3],
  "strategy_key": "percentile",
  "strategy_version": "v1.0",
  "param_space": {
    "window_days": [20, 60],
    "buy_threshold": [0.2, 0.3, 0.4],
    "sell_threshold": [0.8, 0.9]
  },
  "start_date": "2023-01-01",
  "end_date": "2024-01-01",
  "initial_cash": 100000.00,
  "cost_method": "AVERAGE",
  "fee_model": {
    "buy_fee_rate": 0.000845,
    "sell_fee_rate": 0.000845,
    "min_fee": 0.20
  }
}
```

#### 6.2.2 BacktestRun（回测运行）

**职责**：执行一次回测，记录运行状态和核心指标。

**运行流程**：
1. 从 `backtest_plan` 读取配置
2. 根据 `param_space` 生成参数组合（param_set_id）
3. Python服务执行回测（读取历史行情，模拟交易）
4. 计算核心指标（总收益、年化收益、最大回撤、夏普比率、胜率等）
5. 生成明细结果文件（每日曲线、交易明细）存文件系统
6. 摘要指标存 `backtest_run` 表

**核心指标**：
- `total_return`（总收益率）：整个回测期间的总收益率
  - **专有名词公式**：`total_return = (最终市值 - 初始资金) / 初始资金`
  - **解释公式**：总收益率 = (最终值多少钱 - 初始投入多少钱) / 初始投入多少钱
  - **通俗解释**：整个回测期间赚了或亏了多少百分比
- `annual_return`（年化收益率）：按年计算的收益率，用于比较不同时间长度的回测
  - **专有名词公式**：`annual_return = ((1 + total_return) ^ (365 / 回测天数)) - 1`
  - **解释公式**：年化收益率 = ((1 + 总收益率) ^ (365 / 回测天数)) - 1
  - **通俗解释**：如果按这个收益率持续一年，能赚多少
- `max_drawdown`（最大回撤）：回测期间从最高点下跌的最大幅度，反映风险
  - **专有名词公式**：`max_drawdown = MAX((peak_value - current_value) / peak_value)`
  - **解释公式**：最大回撤 = 最大值((峰值 - 当前值) / 峰值)
  - **通俗解释**：回测期间从最高点下跌的最大幅度，负数表示下跌
- `sharpe_ratio`（夏普比率）：风险调整后的收益指标，越高越好
  - **专有名词公式**：`sharpe_ratio = (年化收益率 - 无风险利率) / 年化波动率`
  - **解释公式**：夏普比率 = (年化收益率 - 无风险利率) / 年化波动率
  - **通俗解释**：衡量每承担一单位风险能获得多少收益，越高越好
- `trade_count`（成交次数）：回测期间的总交易次数
- `win_rate`（胜率）：盈利交易占总交易的比例
  - **专有名词公式**：`win_rate = 盈利交易数 / 总交易数`
  - **解释公式**：胜率 = 盈利交易数 / 总交易数
  - **通俗解释**：如0.65表示65%的交易是盈利的
- `total_fees`（手续费总额）：回测期间支付的手续费总和
- `fee_ratio`（手续费占收益比例）：手续费占收益的比例
  - **专有名词公式**：`fee_ratio = total_fees / total_return`
  - **解释公式**：手续费占收益比例 = 手续费总额 / 总收益率

#### 6.2.3 Compare（结果对比）

**对比维度**：
- 同 plan 下的多个 run 结果
- 按收益/回撤/波动/胜率/最大回撤期排序
- 支持多维度筛选和对比

**对比查询**：
```sql
SELECT 
  run_code,
  param_set_id,
  total_return,
  annual_return,
  max_drawdown,
  sharpe_ratio,
  win_rate,
  trade_count
FROM backtest_run
WHERE plan_id = ?
  AND status = 'COMPLETED'
ORDER BY annual_return DESC, max_drawdown ASC;
```

#### 6.2.4 Promote（参数推广）

**流程**：
1. 用户选择最佳 run（基于对比结果）
2. 系统提取 `param_set_id` 和 `param_json`
3. 生成推广预览（显示将更新的配置）
4. 用户二次确认
5. 写入 `strategy_config` 和 `product_strategy_bind`

**推广接口**：
```
POST /api/v2/backtest/promote
{
  "runId": 123,
  "targetProductIds": [1, 2, 3],
  "confirm": true
}
```

### 6.3 回测结果存储策略

**摘要指标（存DB）**：
- 核心指标：总收益、年化收益、最大回撤、夏普比率、胜率等
- 存储位置：`backtest_run` 表

**明细数据（存文件）**：
- 每日曲线：日期、净值、持仓市值、现金余额、总资产
- 交易明细：日期、买卖方向、产品、价格、份额、金额、手续费
- 文件格式：csv.gz 或 parquet
- 存储路径：`artifact_path` 字段记录文件路径

**清理策略**：
- 结果文件TTL：默认保留90天
- 定时任务：清理过期文件
- 用户可手动清理或延长保留期

### 6.4 回测与生产隔离机制

**数据隔离**：
- 回测只读：`market_bar_daily`、`nav`、`indicator_daily`（历史数据）
- 回测不写：`ledger_posting`、`holdings_snapshot`、`net_worth_snapshot`（生产真相表）
- 回测独立表：`backtest_plan`、`backtest_run`（不影响生产）

**执行隔离**：
- Python服务独立进程执行回测（通过ProcessBuilder调用）
- 回测结果不影响生产建议生成
- 参数推广需二次确认，避免误操作

**职责边界**：
- Java职责：回测计划管理、结果对比、参数推广（需二次确认）

- 最佳参数回写：将最佳参数集写回`strategy_config`和`product_strategy_bind`（必须二次确认）

## 10. 建议/信号模型与生命周期

### 8.1 建议生命周期状态流转

```
生成 (ACTIVE) 
  ↓
[用户操作]
  ├─→ 已执行 (EXECUTED) → 关联订单ID
  ├─→ 已忽略 (IGNORED)
  └─→ 过期 (EXPIRED) → 定时任务自动标记
```

### 8.2 去重策略

**唯一键约束**：`uk_user_product_date_strategy` (`user_id`, `product_id`, `generated_at`, `strategy_code`)

- 同产品、同策略、同日只保留最新一条建议
- `generated_at` 使用日期部分（忽略时分秒），确保同一天同一策略只生成一条
- 如果策略在同一天多次运行，新建议会覆盖旧建议（通过 `ON DUPLICATE KEY UPDATE` 实现）

### 8.3 过期管理

**过期策略**：
- `expires_at` 字段：建议有效期（默认生成后24小时）
- 定时任务：每天凌晨检查并标记过期建议
  ```sql
  UPDATE suggestions 
  SET status='EXPIRED' 
  WHERE status='ACTIVE' AND expires_at < NOW();
  ```

### 8.4 复盘最小字段集

为了支持未来复盘，建议表必须保留以下字段：

| 字段 | 用途 |
|------|------|
| `reason` | 触发理由（中文可解释） |
| `reason_blocks_json` | 结构化原因列表（用于详细分析） |
| `trigger_indicators_json` | 触发指标（分位、回撤、溢价率等） |
| `suggest_amount` / `suggest_shares` | 建议金额/份额 |
| `priority` / `confidence` | 优先级/信心度 |
| `generated_at` | 生成时间 |
| `status` / `executed_at` | 执行状态和时间 |

**复盘查询示例**：
```sql
-- 查询某产品过去30天的建议及执行情况
SELECT 
  suggestion_type, reason, priority, confidence,
  generated_at, status, executed_at, executed_order_id
FROM suggestions
WHERE user_id = ? AND product_id = ? 
  AND generated_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
ORDER BY generated_at DESC;
```

### 8.5 建议历史保留策略

- **活跃建议**：保留所有 `ACTIVE` 状态的建议
- **已执行建议**：保留至少90天（用于复盘）
- **已忽略/过期建议**：保留30天后自动清理（或仅保留摘要）

**清理任务**（可选）：
```java
@Scheduled(cron = "0 0 3 * * ?") // 每天凌晨3点
public void cleanupExpiredSuggestions() {
    // 清理30天前的已忽略/过期建议（保留摘要字段）
    LocalDate cutoffDate = LocalDate.now().minusDays(30);
    suggestionMapper.cleanupOldSuggestions(cutoffDate);
}
```

## 11. 看板首页数据口径

### 9.1 今日建议清单（Today Actions）

**数据来源**：`suggestions` 表

**查询条件**：
```sql
SELECT * FROM suggestions
WHERE user_id = ? 
  AND status = 'NEW'
  AND suggestion_day = CURDATE()
  AND (expires_at IS NULL OR expires_at >= NOW())
ORDER BY 
  CASE priority 
    WHEN 'HIGH' THEN 1 
    WHEN 'MEDIUM' THEN 2 
    WHEN 'LOW' THEN 3 
  END,
  generated_at DESC
LIMIT 20;
```

**展示内容（简化版）**：
- **操作内容**（核心展示）：
  - 建议类型图标 + 产品名称（如："买入 纳指ETF"）
  - 建议动作摘要（从 `action_payload` 提取，如："建议买入 5000元" 或 "建议卖出 1000份"）
  - 优先级标签（HIGH/MEDIUM/LOW）
  - 信心度（如：85%）
- **状态标识**：
  - NEW：新建议（可操作）
  - DONE：已执行（显示关联订单）
  - SNOOZE：稍后再看
  - IGNORE：已忽略
- **操作按钮**：
  - 点击操作内容：弹出完整详情（见下方）
  - 如果已下单：显示"确认成交"按钮（见下方）
  - 其他：忽略/稍后再看

**点击操作内容弹出完整详情**：
- 产品完整信息（产品代码、名称、当前净值/价格）
- 完整触发理由（`reason` 字段）
- 详细指标快照（`reason_payload` JSON展开）
- 建议动作参数（`action_payload` JSON展开：金额/目标仓位/阈值等）
- 策略信息（策略代码、版本、生成时间）
- 操作按钮：
  - 如果未下单：显示"去下单"按钮（跳转到外部交易平台或记录订单）
  - 如果已下单：显示"确认成交"按钮（见下方）

**如果已下单，在建议处进行确认/取消/修改**：
- **场景**：用户看到建议后，已在外部平台下单，现在需要在系统中录入成交信息
- **操作流程**：
  1. 点击建议的操作内容，弹出完整详情
  2. 如果检测到用户已下单（通过订单号或手动标记），显示"确认成交"按钮
  3. 点击"确认成交"，弹出成交信息录入弹窗：
     - 订单类型（买入/卖出/申购/赎回）
     - 成交价格/净值（可编辑，默认使用建议中的预期值）
     - 成交份额（买入时）或成交金额（卖出时）
     - 手续费（可编辑）
     - 成交日期（可编辑，默认今日）
     - 确认日期（T+N产品，可编辑）
     - 备注
  4. 用户确认后：
     - 创建 `orders` 记录（订单意图）
     - 如果是T+N产品：创建 `settlement_confirm` 记录（结算确认）
     - 生成 `ledger_txn` 和 `ledger_posting`（流水）
     - 更新 `suggestions.status = 'DONE'`，关联 `executed_order_id`
  5. 如果用户选择"取消"或"修改"：
     - 取消：标记建议为已忽略，不生成订单和流水
     - 修改：允许用户修改成交信息后再确认

**权限过滤**：
- 管理员：可查看全家建议（`family_id` 聚合）
- 普通成员：仅查看自己的建议

### 9.2 待结算清单（Pending Settlement）

**数据来源**：`orders` 表 + `settlement_confirm` 表（LEFT JOIN）

**查询条件**：
```sql
SELECT 
  o.*,
  pm.product_name,
  pm.product_code,
  sc.confirm_date AS actual_confirm_date
FROM orders o
INNER JOIN product_master pm ON o.product_id = pm.id
LEFT JOIN settlement_confirm sc ON o.order_id = sc.order_id
WHERE o.user_id = ? 
  AND o.status = 'PENDING'
ORDER BY o.expected_confirm_date ASC;
```

**展示内容**：
- 产品名称、订单类型、订单金额/份额
- 预期确认日期、预期净值日期
- 预估份额/金额（基于预期净值）
- 实际确认日期（如已确认）
- "确认结算"按钮

**预览净值计算**：
- 买入/申购：`预估份额 = 订单金额 / 预期净值`
- 赎回：`预估金额 = 订单份额 * 预期净值`

### 9.3 今日收益与资产概览

#### 9.3.1 总资产/净资产

**数据来源**：`net_worth_snapshot` 表（最新快照）

**查询条件**：
```sql
SELECT * FROM net_worth_snapshot
WHERE user_id = ? 
  AND snapshot_date = (
    SELECT MAX(snapshot_date) 
    FROM net_worth_snapshot 
    WHERE user_id = ?
  );
```

**展示内容**：
- 总资产、净资产、现金余额、持仓市值
- 今日盈亏 = 今日净资产 - 昨日净资产
- 累计收益 = 当前净资产 - 初始投入（从流水汇总）

#### 9.3.2 今日盈亏计算

**数据来源**：`holdings_snapshot` 表（对比昨日）

```sql
-- 今日持仓市值
SELECT SUM(market_value) FROM holdings_snapshot
WHERE user_id = ? AND snapshot_date = CURDATE();

-- 昨日持仓市值
SELECT SUM(market_value) FROM holdings_snapshot
WHERE user_id = ? AND snapshot_date = DATE_SUB(CURDATE(), INTERVAL 1 DAY);

-- 今日盈亏 = 今日市值 - 昨日市值 + 今日现金变动
```

#### 9.3.3 XIRR（资金加权收益率）计算

**数据来源**：`ledger_posting` 表（现金流）

**计算逻辑**（Python计算，Java调用）：
- 提取所有投资类交易的现金流（买入为负，卖出为正，分红为正）
- 使用 XIRR 算法计算年化收益率
- 返回结果：XIRR值、累计收益、年化收益

### 9.4 资产配置

**数据来源**：`holdings_snapshot` 表 + `accounts` 表

**按资产类别汇总**：
```sql
SELECT 
  pm.asset_type,
  SUM(hs.market_value) AS total_value,
  SUM(hs.market_value) / (SELECT SUM(market_value) FROM holdings_snapshot WHERE user_id = ?) AS weight
FROM holdings_snapshot hs
INNER JOIN product_master pm ON hs.product_id = pm.id
WHERE hs.user_id = ? AND hs.snapshot_date = CURDATE()
GROUP BY pm.asset_type;
```

**按账户汇总**：
```sql
SELECT 
  a.account_type,
  SUM(a.balance) AS total_balance
FROM accounts a
WHERE a.owner_user_id = ? OR (a.owner_family_id = ? AND a.owner_type = 'FAMILY')
GROUP BY a.account_type;
```

## 14. 验收标准与示例数据流

### 12.1 场外基金买入完整流程

#### 场景：用户买入场外基金 5000元，T+1确认

**步骤1：创建订单（下单阶段：占用资金，不扣款）**
```
POST /api/v2/orders
{
  "productId": 1,
  "orderType": "SUBSCRIPTION",
  "amount": 5000.00,
  "requestedAt": "2024-01-15 14:30:00"
}

系统处理：
1. 创建 orders 记录
   - order_id: "ORD-20240115-001"
   - status: "PENDING"
   - trade_date: "2024-01-15" (14:30 < 15:00 cutoff)
   - expected_nav_date: "2024-01-15"
   - expected_confirm_date: "2024-01-16" (T+1)

2. 资金占用（关键！不生成流水）
   - accounts.reserved_amount += 5000.00（占用增加）
   - accounts.balance 不变（此时还未真正扣款）
   - available_for_trade = balance - reserved_amount

3. 不生成 ledger_txn 和 ledger_posting（待结算确认时再生成）
```

**验证**：
- **初始状态计算**：
  - balance = 10000元（账面余额）
  - reserved = 0元（占用金额）
  - **专有名词公式**：`available_for_trade = balance - reserved_amount`
  - **解释公式**：可用购买力 = 账面余额 - 占用金额
  - **具体计算**：available = 10000 - 0 = 10000元
- **下单后状态计算**：
  - balance = 10000元（不变，因为还未真正扣款）
  - reserved = 5000元（占用增加）
  - **具体计算**：available = 10000 - 5000 = 5000元 ✓

**步骤2：待结算清单显示**
```
GET /api/v2/dashboard/pending-settlements

返回：
[
  {
    "orderId": "ORD-20240115-001",
    "productName": "易方达沪深300ETF联接A",
    "orderType": "SUBSCRIPTION",
    "amount": 5000.00,
    "expectedConfirmDate": "2024-01-16",
    "expectedNavDate": "2024-01-15",
    "previewShares": 4854.37,  // 5000 / 1.03 (预估净值)
    "status": "PENDING"
  }
]
```

**步骤3：确认结算（释放占用并生成真实流水）**
```
POST /api/v2/settlements/confirm
{
  "orderId": "ORD-20240115-001",
  "confirmDate": "2024-01-16",
  "navDate": "2024-01-15",
  "confirmNav": 1.0305,  // 实际净值（用户核对后输入）
  "confirmShares": 4851.55,  // 实际份额（用户核对后输入）
  "confirmFee": 4.25,
  "isManualOverride": true  // 用户手动修正
}

系统处理：
1. 创建 settlement_confirm 记录
   - order_id: "ORD-20240115-001"
   - confirm_date: "2024-01-16"
   - nav_date: "2024-01-15"
   - confirm_nav: 1.0305
   - confirm_shares: 4851.55
   - is_manual_override: true

2. 释放占用
   - accounts.reserved_amount -= 5000.00（释放占用）

3. 生成真实流水（此时才真正扣款）
   - 生成 ledger_txn（txn_type='SUBSCRIPTION', status='CONFIRMED'）
   - 生成 ledger_posting（CREDIT CASH 5004.25，现金减少，实际扣款总额 = 5000 + 4.25）
   - 生成 ledger_posting（DEBIT POSITION shares=4851.55, amount=5000.00，持仓增加，成本口径不含手续费）
   - 生成 ledger_posting（DEBIT FEE amount=4.25，手续费）

4. 更新账户余额
   - **余额计算**：
     - **专有名词公式**：`balance = initial_balance + SUM(DEBIT金额) - SUM(CREDIT金额)`
     - **解释公式**：余额 = 初始余额 + 所有增加金额 - 所有减少金额
     - **具体计算**：
       - 初始余额：10000元
       - 增加金额：0元（没有收入）
       - 减少金额：5004.25元（实际扣款总额 = 5000 + 4.25手续费）
       - 余额 = 10000 + 0 - 5004.25 = 4995.75元（注意：实际扣款是5004.25，不是5000）
   - 或者简化为：accounts.balance = 10000 - 5004.25 = 4995.75（此时才真正扣款）

5. 更新订单状态
   - orders.status = "CONFIRMED"

**验证**：
- **确认前状态**：
  - balance = 10000元，reserved = 5000元
  - **专有名词公式**：`available_for_trade = balance - reserved_amount`
  - **具体计算**：available = 10000 - 5000 = 5000元
- **确认后状态**：
  - balance = 4995.75元（扣款5004.25元），reserved = 0元（释放占用）
  - **具体计算**：available = 4995.75 - 0 = 4995.75元 ✓
- **持仓计算**：
  - **专有名词公式**：`shares = SUM(CASE WHEN posting_type='DEBIT' THEN shares ELSE -shares END)`
  - **具体计算**：持仓增加 = +4851.55 shares ✓
```

### 12.2 账户余额手工调整流程

#### 场景：用户发现现金账户余额不对，手工调整为10000元

**步骤1：调整余额**
```
PUT /api/v2/accounts/1/balance
{
  "newBalance": 10000.00,
  "note": "对账发现差异，手工调整"
}

系统处理：
1. 查询当前余额：accounts.balance = 8500.00
2. **计算差异**：
   - 目标余额：10000.00元
   - 当前余额：8500.00元
   - **差异计算**：差异 = 10000.00 - 8500.00 = 1500.00元
   - **为什么需要调整分录**：余额调整必须自动生成流水，保证可追溯，不允许直接修改余额
3. 生成 ledger_txn
   - txn_id: "TXN-20240115-ADJ-001"
   - txn_type: "ADJUST"（ADJUST，即余额调整）
   - note: "对账发现差异，手工调整"

4. 生成 ledger_posting（余额增加）
   - posting_type: "DEBIT"（DEBIT，对于资产类账户表示增加）
   - account_id: 1
   - account_type: "CASH"（CASH，即现金账户）
   - amount: 1500.00

5. **更新账户余额**：
   - **专有名词公式**：`balance = initial_balance + SUM(DEBIT金额) - SUM(CREDIT金额)`
   - **解释公式**：余额 = 初始余额 + 所有增加金额 - 所有减少金额
   - **具体计算**：
     - 调整前余额：8500.00元
     - 调整分录（DEBIT）：+1500.00元
     - 调整后余额：8500.00 + 1500.00 = 10000.00元 ✓
   - 或者直接更新：accounts.balance = 10000.00（由posting推导，或直接更新后通过对账检查验证）
```

**关键点**：余额调整必须自动生成流水，保证可追溯。

### 12.3 账户间转账流程

**步骤1：创建转账**
```
POST /api/v2/ledger/txns
{
  "txnType": "TRANSFER_OUT",
  "fromAccountId": 1,  // 银行账户
  "toAccountId": 2,    // 券商账户

   - note: "从银行账户转账到券商账户"

2. 生成 ledger_posting（银行账户减少）
   - posting_type: "CREDIT"
   - account_id: 1
   - account_type: "CASH"
   - amount: 5000.00

   - txn_id: "TXN-20240115-TRF-002"
   - txn_type: "TRANSFER_IN"
   - biz_group_key: "TRF-20240115-001"  // 同一业务分组

   - posting_type: "DEBIT"
   - account_id: 2
   - account_type: "CASH"
   - amount: 5000.00

5. **更新账户余额**：
   - **账户1（银行账户）余额计算**：

     - **具体计算**：balance = balance + 5000.00（DEBIT增加，即转入）
```

#### 场景：基金分红，除权日2024-01-15，派息日2024-01-20，每股分红0.1元，持仓1000股

**步骤1：除权日（EX_DATE）**
```
系统处理：
1. 生成 ledger_txn
   - txn_id: "TXN-20240115-DIV-EX-001"
   - txn_type: "DIVIDEND_EX_DATE"
   - trade_date: "2024-01-15"
   - nav_date: "2024-01-15"

   - DEBIT RECEIVABLE 100.00（应收股利增加 = 1000股 × 0.1元）

3. 生成 ledger_posting（贷：投资收益）

   - 持仓市值相应下跌100元

5. **净资产计算**：
   - **专有名词公式**：`net_worth = total_assets - total_liabilities`
   - **解释公式**：净资产 = 总资产 - 总负债

     - 净资产不变 ✓
   - **为什么这样算**：除权日净值下跌，但应收股利增加，总资产不变，所以净资产不变
```

```java
/**
 * 计算已实现盈亏
 * 
 * 专有名词公式：realized_pnl = (卖出净到账金额) - (卖出部分成本) - (手续费)
 * 解释公式：已实现盈亏 = 卖出后实际收到的钱 - 卖出那部分的本金 - 手续费
 * 
 * @param sellShares 卖出份额
 * @param sellNav 卖出净值（每份价格）
 * @param sellFee 手续费
 * @param avgCost 平均成本（每份成本）
 * @return 已实现盈亏（正数表示盈利，负数表示亏损）
 * 
 * 计算步骤：
 * 1. 计算卖出总金额：sellShares × sellNav
 * 2. 计算卖出净到账：卖出总金额 - sellFee
 * 3. 计算卖出部分成本：sellShares × avgCost
 * 4. 计算已实现盈亏：卖出净到账 - 卖出部分成本
 * 
 * 为什么这样算：
 * - 卖出净到账是"实际收到的钱"
 * - 卖出部分成本是"本钱"（当初买入花的钱）
 * - 手续费是"交易成本"
 * - 只有超过本钱和手续费的部分才是"赚的"
 */
public BigDecimal calculateRealizedPnl(BigDecimal sellShares, BigDecimal sellNav, 
                                       BigDecimal sellFee, BigDecimal avgCost) {
    // 计算卖出总金额
    BigDecimal sellTotal = sellShares.multiply(sellNav);
    
    // 计算卖出净到账（实际收到的钱）
    BigDecimal sellNetAmount = sellTotal.subtract(sellFee);
    
    // 计算卖出部分成本（本钱）
    BigDecimal sellCost = sellShares.multiply(avgCost);
    
    // 计算已实现盈亏（赚的钱）
    return sellNetAmount.subtract(sellCost);
}
```

### 16.2 日志设计策略

#### 16.2.1 日志级别定义

**日志级别分级**（按必要性和重要性）：
1. **ERROR**：错误日志，必须记录
   - 系统异常、计算错误、数据不一致等
   - 必须保留，用于问题排查
2. **WARN**：警告日志，重要记录
   - 数据异常但可恢复、业务规则违反等
   - 建议保留，用于问题预警
3. **INFO**：信息日志，选择性记录
   - 关键业务操作（下单、结算、余额调整等）
   - 计算公式执行过程（可选，调试时开启）
   - 按需保留，用于业务追踪
4. **DEBUG**：调试日志，临时记录
   - 详细计算过程、中间变量值等
   - 仅开发/调试时使用，生产环境关闭
   - 不保留，避免占用空间

#### 16.2.2 日志内容规范

**必须记录的日志**：
1. **计算错误日志**：
   - 记录计算公式、输入参数、计算结果、错误原因
   - 格式：`[计算错误] 公式={公式}, 输入={参数}, 输出={结果}, 错误={原因}`

2. **数据不一致日志**：
   - 记录预期值、实际值、差异、可能原因
   - 格式：`[数据不一致] 字段={字段}, 预期={预期值}, 实际={实际值}, 差异={差异}`

3. **关键业务操作日志**：
   - 记录操作类型、操作对象、操作结果
   - 格式：`[业务操作] 类型={类型}, 对象={对象}, 结果={结果}`

**可选记录的日志**：
1. **计算公式执行日志**（INFO级别，可配置开关）：
   - 记录公式、输入、输出、计算步骤
   - 格式：`[公式计算] 公式={公式}, 输入={参数}, 步骤={步骤}, 输出={结果}`

2. **性能监控日志**（INFO级别，可配置开关）：
   - 记录操作耗时、数据量等
   - 格式：`[性能监控] 操作={操作}, 耗时={耗时}ms, 数据量={数量}`

**不记录的日志**：
1. **详细计算过程**（DEBUG级别，生产环境关闭）
2. **中间变量值**（DEBUG级别，生产环境关闭）
3. **频繁的操作日志**（如每次查询余额，除非异常）

## 17. 最小验算用例（必须通过）

### 用例1：扣现金永远不会让现金余额增加

**初始状态**：
- 现金账户：balance = 10000, reserved = 0

**操作1：下单买入5000元（占用阶段）**
- reserved += 5000（占用金额增加）
- balance 不变（此时还未真正扣款）
- **验证**：
  - **专有名词公式**：`available_for_trade = balance - reserved_amount`
  - **解释公式**：可用购买力 = 账面余额 - 占用金额
  - **具体计算**：available = 10000 - 5000 = 5000 ✓

**操作2：结算确认（扣款阶段）**
- 生成 ledger_posting：CREDIT CASH 5000（现金减少）
- reserved -= 5000（释放占用）
- **余额计算**：
  - **专有名词公式**：`balance = initial_balance + SUM(DEBIT金额) - SUM(CREDIT金额)`
  - **解释公式**：余额 = 初始余额 + 所有增加金额 - 所有减少金额
  - **具体计算**：balance = 10000 + 0 - 5000 = 5000
- **验证**：balance = 5000, reserved = 0, available = 5000 ✓

**断言**：扣现金操作永远导致余额减少，不会增加 ✓

### 用例2：转账账户合计不变

**初始状态**：
- 账户A：balance = 10000
- 账户B：balance = 5000

**操作：转账1000元（A → B）**
- 生成 ledger_posting：CREDIT A账户 1000（A减少）
- 生成 ledger_posting：DEBIT B账户 1000（B增加）
- **验证**：
- **账户A余额计算**：
  - **专有名词公式**：`balance = initial_balance + SUM(DEBIT金额) - SUM(CREDIT金额)`
  - **具体计算**：balance = 10000 + 0 - 1000 = 9000 ✓
- **账户B余额计算**：
  - **具体计算**：balance = 5000 + 1000 - 0 = 6000 ✓
- **合计验证**：9000 + 6000 = 15000（不变）✓

**断言**：转账操作账户合计不变 ✓

### 用例3：信用卡消费负债增加

**初始状态**：

**操作：信用卡消费1000元**

    - **解释公式**：负债余额 = 初始负债 + 所有增加负债的金额总和 - 所有减少负债的金额总和
    - **具体计算**：balance = 0 + 1000 - 0 = 1000（欠款1000）✓
  - **费用账户余额计算**：
    - **专有名词公式**：`balance = SUM(DEBIT金额) - SUM(CREDIT金额)`
    - **解释公式**：费用总额 = 所有发生的费用金额总和 - 所有冲减的费用金额总和
    - **具体计算**：balance = 1000 - 0 = 1000（费用1000）✓

**操作：还款333.33元（本金）**

## 附录：命名规范建议

### 表名规范
- 使用小写字母 + 下划线：`ledger_txn`, `product_master`
- 复数形式：`users`, `accounts`, `orders`
- 单数形式（主数据）：`product_master`, `user_product`

### 字段名规范
- 使用小写字母 + 下划线：`user_id`, `product_code`, `confirm_date`
- 布尔字段：`is_active`, `is_manual_override`
- 日期字段：`created_at`, `updated_at`, `trade_date`, `nav_date`

### 接口路径规范
- RESTful风格：`/api/v2/{resource}/{id}`
- 动作使用动词：`/api/v2/orders/{id}/cancel`, `/api/v2/settlements/confirm`
- 查询使用查询参数：`?status=&startDate=&endDate=`

### 枚举值规范
- 使用大写字母 + 下划线：`PENDING`, `CONFIRMED`, `ACTIVE`, `EXECUTED`
- 类型枚举：`BUY`, `SELL`, `SUBSCRIPTION`, `REDEMPTION`
