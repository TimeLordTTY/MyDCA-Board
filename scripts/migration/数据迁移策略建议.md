# 数据迁移策略建议

## 一、V1与V2核心差异分析

### 1.1 账户系统（accounts）
**V1结构：**
- `account_type`: CASH/BUCKET/FUND_MAPPED/PRODUCT_SUB/FUND_TOTAL/SUMMARY
- `parent_account_id`: 父子关系
- `shares`: 直接存储在产品子账户中

**V2结构：**
- `account_kind`: REAL/VIRTUAL（现实账户 vs 虚拟科目）
- `virtual_subtype`: POSITION/FEE/INCOME/EXPENSE/RECEIVABLE/LIABILITY
- `reserved_amount`: 占用/冻结金额（V1没有）
- `parent_account_id`: 父子关系（V2重新支持，父=分组容器、子=真实信封余额）
- `fund_usage`: 资金用途（SPENDABLE/RESERVED/INVESTABLE，V1没有）

**迁移难点：**
- V1的`BUCKET`类型账户 => V2的REAL父子账户（更直接）
- **重要**：说明V1的BUCKET与V2的VIRTUAL科目不是一回事：BUCKET属于REAL账户的用途分区
- **重要**：V2不再需要"父余额=真实余额"的迁移逻辑（父余额由子求和展示，父账户不参与记账）
- V1的"平台分区子账户余额" => V2的叶子账户余额（直接迁移）
- V1的`PRODUCT_SUB`类型账户需要转换为V2的`VIRTUAL POSITION`账户
- V1的`FUND_MAPPED`/`FUND_TOTAL`/`SUMMARY`需要重新设计映射规则
- `reserved_amount`在V1中不存在，需要从历史订单推导
- `fund_usage`字段的迁移规则：按V1用途推导（生活费=SPENDABLE，房租/项目/安全金=RESERVED，理财金=INVESTABLE）

### 1.2 流水系统（ledger）
**V1结构：**
- 单表`ledger`，包含`entry_type`, `amount`, `account_from`, `account_to`等字段
- 简单的一对一或一对多关系

**V2结构：**
- 双分录：`ledger_txn`（事件头）+ `ledger_posting`（分录明细）
- 严格的借贷方向规则：`amount`永远为正数，`posting_type`决定方向
- `biz_group_key`用于UI合并展示

**迁移难点：**
- V1的单条记录需要拆分为V2的一条`txn` + 多条`posting`
- 需要根据V1的`entry_type`和`account_from`/`account_to`推导借贷方向
- V1可能没有明确的`txn_id`，需要生成
- V1的`amount`可能是负数，需要转换为正数+方向

### 1.3 订单系统（orders）
**V1结构：**
- `orders`表包含`amount`, `shares`, `fee`, `nav`, `confirm_date`等完整信息
- 状态字段较简单

**V2结构：**
- `orders`表只存订单意图（`amount`/`shares`为预期值）
- `settlement_confirm`表独立存储结算确认结果（`confirm_nav`, `confirm_shares`, `confirm_amount`等）
- 新增`expected_confirm_date`和`reserved_amount`机制

**迁移难点：**
- V1的已确认订单需要拆分为`orders` + `settlement_confirm`
- V1的`nav`/`confirm_date`需要迁移到`settlement_confirm`
- V1的`status`需要映射到V2的状态枚举

### 1.4 持仓快照
**V1结构：**
- `daily_snapshot`：按产品+日期存储持仓、成本、盈亏等
- `daily_balance`：按账户+日期存储余额

**V2结构：**
- `holdings_snapshot`：按用户+产品+日期存储持仓快照
- `net_worth_snapshot`：按用户+日期存储净资产快照
- 新增`is_dirty`和`dirty_from_date`用于历史修正重算

**迁移难点：**
- V1的`daily_snapshot`需要按用户拆分（V1可能没有用户概念）
- V1的`product_code`需要映射到V2的`product_id`
- V1的快照数据可能需要重新计算以符合V2的借贷规则

### 1.5 用户权限系统
**V1结构：**
- 可能没有明确的用户表，或只有简单的用户概念

**V2结构：**
- `users` + `families` + `user_family_roles`
- 账户需要关联`owner_user_id`或`owner_family_id`

**迁移难点：**
- 如果V1没有用户概念，需要创建默认用户和家庭
- 所有账户、订单、流水都需要关联到用户/家庭

### 1.6 其他差异
- **策略配置**：V1的`strategy_config`和`product_strategy_bind`结构基本一致，可直接迁移
- **市场数据**：V1的`market_bar_d`/`market_quote_rt`/`nav`与V2的`market_bar_daily`/`market_quote_realtime`/`nav_daily`基本一致，表名和字段名略有差异
- **回测数据**：V1的`backtest_summary`/`backtest_daily`/`backtest_trades`与V2的`backtest_plan`/`backtest_run`结构差异较大，需要重新设计映射

## 二、迁移方案对比

### 方案A：现在迁移（开发前迁移）

#### 优点：
1. **数据验证早**：可以在开发过程中用真实数据测试系统
2. **问题发现早**：迁移脚本的问题可以在开发阶段发现和修复
3. **开发有参考**：开发时可以参考真实数据的格式和规模
4. **并行工作**：迁移脚本开发和系统开发可以并行进行

#### 缺点：
1. **迁移脚本复杂**：需要处理大量字段映射和业务逻辑转换
2. **可能多次调整**：如果V2设计在开发过程中调整，迁移脚本也需要同步调整
3. **数据不一致风险**：迁移后的数据可能不符合V2的业务规则，需要大量校验和修复
4. **开发干扰**：迁移脚本的调试可能占用开发时间

#### 适用场景：
- V2设计已经非常稳定，不会再有大的调整
- 有充足时间编写和测试迁移脚本
- 需要真实数据来指导开发

---

### 方案B：开发完成后迁移（推荐）

#### 优点：
1. **设计稳定**：V2系统开发完成后，表结构和业务规则已经稳定，迁移脚本不会因为设计变更而反复修改
2. **迁移脚本更准确**：基于最终稳定的V2设计编写，减少返工
3. **数据校验更完善**：V2系统已经实现了数据校验逻辑，迁移后的数据可以直接用系统功能验证
4. **开发专注**：开发阶段专注于功能实现，不需要考虑数据兼容性
5. **迁移工具化**：可以利用V2系统的API和工具来辅助迁移，而不是直接写SQL

#### 缺点：
1. **测试数据有限**：开发阶段只能用模拟数据，可能遗漏一些边界情况
2. **迁移压力集中**：迁移工作集中在系统上线前，时间压力较大
3. **问题发现晚**：迁移过程中的问题可能影响上线时间

#### 适用场景：
- V2设计可能在开发过程中有调整
- 希望开发阶段专注于功能实现
- 有足够时间在开发完成后进行迁移

---

## 三、推荐方案：**开发完成后迁移（方案B）**

### 3.1 推荐理由

1. **设计稳定性**：你的V2系统是完全重构，设计文档虽然已经完善，但在实际开发过程中可能还会发现需要调整的地方。如果现在迁移，迁移脚本可能需要反复修改。

2. **迁移复杂度高**：V1到V2的差异非常大，特别是：
   - 单表`ledger` → 双分录`ledger_txn` + `ledger_posting`
   - 账户结构完全不同
   - 订单和结算分离
   - 新增用户权限系统
   
   这些差异需要复杂的业务逻辑转换，如果V2设计还在调整，迁移脚本会非常难维护。

3. **数据校验依赖系统**：V2系统有严格的数据校验规则（如借贷方向、余额一致性等），迁移后的数据需要这些校验来保证正确性。如果现在迁移，你需要自己实现这些校验逻辑，工作量很大。

4. **工具化迁移**：开发完成后，你可以：
   - 利用V2的API来写入数据（而不是直接写SQL），这样可以利用系统的校验逻辑
   - 编写Python迁移脚本，调用V2的Java API
   - 或者编写Java迁移工具，复用V2的业务逻辑

### 3.2 迁移实施步骤（开发完成后）

#### 阶段1：迁移准备（1-2天）
1. **数据备份**：完整备份V1数据库
2. **数据统计**：统计V1各表的数据量，评估迁移时间
3. **映射规则设计**：
   - V1账户类型 → V2账户类型映射表
   - V1流水类型 → V2交易类型映射表
   - V1订单状态 → V2订单状态映射表

#### 阶段2：迁移脚本开发（3-5天）
1. **基础数据迁移**：
   - `products` → `product_master`（基本一致，字段映射即可）
   - `categories` → `categories`（基本一致）
   - `strategy_config` → `strategy_config`（基本一致）
   - `product_strategy_bind` → `product_strategy_bind`（基本一致）

2. **用户权限迁移**：
   - 创建默认用户和家庭（如果V1没有用户概念）
   - 所有数据关联到默认用户/家庭

3. **账户迁移**（最复杂）：
   - V1 `accounts` → V2 `accounts`
   - 需要根据V1的`account_type`推导V2的`account_kind`和`virtual_subtype`
   - 计算`reserved_amount`（从历史订单推导）

4. **流水迁移**（最复杂）：
   - V1 `ledger` → V2 `ledger_txn` + `ledger_posting`
   - 需要根据V1的`entry_type`和`account_from`/`account_to`推导借贷方向
   - 生成`txn_id`和`biz_group_key`

5. **订单迁移**：
   - V1 `orders` → V2 `orders` + `settlement_confirm`
   - 已确认订单需要拆分

6. **快照迁移**：
   - V1 `daily_snapshot` → V2 `holdings_snapshot`
   - V1 `daily_balance` → V2 `net_worth_snapshot`（需要汇总）

7. **市场数据迁移**：
   - V1 `market_bar_d` → V2 `market_bar_daily`（字段映射）
   - V1 `market_quote_rt` → V2 `market_quote_realtime`（字段映射）
   - V1 `nav` → V2 `nav_daily`（字段映射）

#### 阶段3：数据校验（2-3天）
1. **余额一致性校验**：
   - 对比V1和V2的账户余额
   - 对比V1和V2的持仓份额和成本

2. **流水完整性校验**：
   - 确保每条V1流水都有对应的V2 `txn`和`posting`
   - 确保借贷平衡（每个`txn`的DEBIT总额 = CREDIT总额）

3. **订单完整性校验**：
   - 确保每个V1订单都有对应的V2订单和结算确认

4. **快照一致性校验**：
   - 对比V1和V2的快照数据

#### 阶段4：问题修复（2-5天）
1. **数据修复**：根据校验结果修复迁移错误
2. **业务规则修复**：确保迁移后的数据符合V2的业务规则
3. **性能优化**：优化迁移脚本性能，确保大数据量迁移的可行性

#### 阶段5：试运行（3-7天）
1. **小批量迁移**：先迁移最近1个月的数据，验证迁移脚本
2. **功能验证**：在V2系统中验证迁移后的数据是否正常
3. **全量迁移**：确认无误后，进行全量迁移

## 四、迁移脚本开发建议

### 4.1 技术选型
- **语言**：Python（你已经有Python环境，且适合数据处理）
- **数据库连接**：使用`pymysql`或`mysql-connector-python`
- **迁移工具**：可以编写独立的Python脚本，也可以利用V2的Java API（通过HTTP调用）

### 4.2 脚本结构建议
```
scripts/migration/
├── config.py              # 数据库连接配置
├── mapping.py             # V1→V2映射规则
├── migrate_base.py        # 基础数据迁移（products, categories等）
├── migrate_accounts.py    # 账户迁移
├── migrate_ledger.py      # 流水迁移（最复杂）
├── migrate_orders.py     # 订单迁移
├── migrate_snapshots.py   # 快照迁移
├── migrate_market.py      # 市场数据迁移
├── validate.py            # 数据校验
└── main.py                # 主入口，按顺序执行迁移
```

### 4.3 关键迁移逻辑示例

#### 账户迁移示例
```python
def migrate_account(v1_account):
    """将V1账户转换为V2账户"""
    v2_account = {
        'account_code': v1_account['account_code'],
        'account_name': v1_account['account_name'],
        'currency': v1_account['currency'],
        'balance': v1_account['balance'],
        'initial_balance': v1_account['balance'],  # 假设初始余额等于当前余额
        'reserved_amount': 0.00,  # 需要从订单推导
        'is_active': v1_account['is_active'],
        'note': v1_account['note'],
    }
    
    # 映射账户类型
    if v1_account['account_type'] == 'CASH':
        v2_account['account_kind'] = 'REAL'
        v2_account['account_type'] = 'CASH'
    elif v1_account['account_type'] == 'PRODUCT_SUB':
        v2_account['account_kind'] = 'VIRTUAL'
        v2_account['virtual_subtype'] = 'POSITION'
        v2_account['account_type'] = 'OTHER'  # 需要根据实际情况调整
    # ... 其他类型映射
    
    return v2_account
```

#### 流水迁移示例
```python
def migrate_ledger_entry(v1_entry):
    """将V1流水转换为V2双分录"""
    # 生成交易事件
    txn = {
        'txn_id': generate_txn_id(v1_entry),
        'txn_type': map_entry_type_to_txn_type(v1_entry['entry_type']),
        'requested_at': v1_entry['event_time'],
        'trade_date': v1_entry['event_time'].date(),
        'status': 'CONFIRMED',
        'note': v1_entry['note'],
    }
    
    # 生成分录
    postings = []
    if v1_entry['entry_type'] == 'expense':
        # 支出：CASH CREDIT, EXPENSE DEBIT
        postings.append({
            'posting_type': 'CREDIT',
            'account_id': get_account_id(v1_entry['account_from']),
            'account_type': 'CASH',
            'amount': abs(v1_entry['amount']),  # 转为正数
        })
        postings.append({
            'posting_type': 'DEBIT',
            'account_id': get_expense_account_id(v1_entry['category_l1']),
            'account_type': 'EXPENSE',
            'amount': abs(v1_entry['amount']),
        })
    # ... 其他类型处理
    
    return txn, postings
```

## 五、风险与应对

### 5.1 主要风险
1. **数据量过大**：如果V1数据量很大，迁移时间可能很长
2. **数据不一致**：V1和V2的业务规则不同，可能导致迁移后数据不符合V2规则
3. **迁移脚本bug**：复杂的迁移逻辑容易出现bug
4. **时间压力**：如果迁移时间过长，可能影响上线时间

### 5.2 应对措施
1. **分批迁移**：按时间或类型分批迁移，降低风险
2. **回滚方案**：准备回滚脚本，如果迁移失败可以快速回滚
3. **充分测试**：在测试环境充分测试迁移脚本
4. **数据校验**：编写完善的数据校验脚本，确保迁移质量
5. **并行开发**：迁移脚本开发和系统开发可以并行进行，但迁移执行放在系统开发完成后

---

## 六、最终建议

**推荐：开发完成后迁移（方案B）**

**理由总结：**
1. V2设计可能在开发过程中调整，现在迁移容易返工
2. 迁移复杂度高，需要稳定的V2设计作为基础
3. 可以利用V2系统的校验逻辑来保证数据质量
4. 开发阶段可以专注于功能实现，不受迁移干扰

**时间安排建议：**
- **开发阶段**：专注于V2系统开发，用模拟数据测试
- **开发完成后**：预留1-2周时间进行数据迁移和验证
- **迁移脚本开发**：可以在开发后期并行进行，但执行放在开发完成后

**如果必须现在迁移：**
- 只迁移基础数据（products, categories, strategy_config等），这些表结构基本一致
- 复杂数据（accounts, ledger, orders）等开发完成后再迁移
- 这样可以部分利用真实数据，同时避免复杂迁移的返工风险
