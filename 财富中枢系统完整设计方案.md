## 2. 领域模型与模块拆分

### 2.1 核心领域模型

#### 2.1.1 用户与权限域
- **用户 (User)**: 系统使用者，属于某个家庭
- **家庭 (Family)**: 多用户组织单元
- **角色 (Role)**: ADMIN（管理员）/ MEMBER（普通成员）
- **权限规则**: 管理员可看全家，成员只看自己+授权部分

#### 2.1.2 产品域
- **产品主数据 (ProductMaster)**: 全局唯一产品（code + channel + market）
- **用户产品配置 (UserProduct)**: 个人对产品的配置（别名、策略绑定、风险标签等）
- **产品类别**: ETF / LOF / FUND / MMF / BANK_WM_NAV / BANK_WM_BOX / STOCK / FUTURES / OPTIONS

#### 2.1.3 账户域
- **账户 (Account)**: 资金账户（银行/支付/券商/现金等）或虚拟科目（持仓/费用/收入/负债等）
- **账户分类**: 
  - **REAL（现实账户）**: 银行/支付/券商/现金等，允许手工余额调整（需生成流水）
  - **VIRTUAL（虚拟科目）**: 持仓/费用/收入/支出/应收/负债等，禁止手工改余额，仅用于分录归集与报表
- **账户归属**: 个人账户 / 家庭共同账户
- **余额口径**:
  - `balance`: 账面余额（由流水推导）
  - `reserved_amount`: 占用/冻结金额（下单占用，结算确认后释放）
  - `available_for_trade`: 可用购买力 = balance - reserved_amount
  - `available_for_withdraw`: 可取资金 = balance - 未交收卖出款 - reserved_amount

#### 2.1.4 交易域（核心）
- **交易事件 (LedgerTxn)**: 业务事件（买入/卖出/赎回/转账/退款/报销/分红等）
- **分录 (LedgerPosting)**: 对账户的借贷影响（金额/份额/费用）
- **订单 (Order)**: 交易意图（发起时间、预期确认日等）
- **结算确认 (SettlementConfirm)**: 实际结算结果（确认日期、实际净值、实际份额/金额）

#### 2.1.5 持仓域
- **持仓 (Holding)**: 由流水推导，可缓存快照
- **持仓快照 (HoldingsSnapshot)**: 每日持仓快照（提升查询性能）
- **净资产快照 (NetWorthSnapshot)**: 每日净资产快照

#### 2.1.6 行情域
- **日线行情 (MarketBarDaily)**: 长期保存
- **实时行情 (MarketQuoteRealtime)**: 短期缓存（7-30天）
- **净值 (Nav)**: 长期保存
- **指标 (IndicatorDaily)**: 日更指标（分位、回撤、均线等）

#### 2.1.7 策略域
- **策略配置 (StrategyConfig)**: 策略参数配置
- **产品策略绑定 (ProductStrategyBind)**: 产品与策略的绑定关系
- **策略状态 (StrategyState)**: 策略运行状态（利润池、峰值等）
- **信号 (Signal)**: 策略生成的原始信号
- **建议 (Suggestion)**: 面向用户的建议（去重、过期、状态管理）

#### 2.1.8 基本面域（三层）
- **层1 - 产品标签 (ProductTag)**: 长期白名单、风险标签、核心等级
- **层2 - 组合目标 (PortfolioTarget)**: 目标权重范围、再平衡阈值
- **层3 - 交易档位 (TradingGear)**: 日/周执行档位开关（油门大小）

#### 2.1.9 回测实验室域（一等模块）
- **回测计划 (BacktestPlan)**: 实验配置（产品范围、策略版本、参数空间、时间范围、成本法、手续费模型）
- **回测运行 (BacktestRun)**: 一次运行实例（plan_id + param_set_id + 状态 + 核心指标 + 结果引用）
- **回测结果**: 摘要指标存DB，明细数据存文件（csv.gz/parquet）
- **参数推广 (Promote)**: 将最佳参数集写回生产配置（需二次确认）

### 2.4 Python任务执行机制

#### 2.4.1 任务分类

**同步任务**（Java等待Python返回）：
- 单只标的查询：`python scripts/market/get_quote.py --product_id 5 --output result.json`
- 轻量建议生成：`python scripts/strategy/generate_suggestions.py --user_id 1 --product_ids [1,2,3] --output suggestions.json`
- 指标计算：`python scripts/indicator/calc_percentile.py --product_id 5 --window_days 60 --output indicator.json`

**异步任务**（Java启动后返回job_id，前端轮询）：
- 回测执行：`python scripts/backtest/run_backtest.py --run_id 123 --input input.json --output output.json`
- 批量行情归档：`python scripts/market/batch_sync.py --product_ids [1,2,3] --start_date 2023-01-01 --end_date 2024-01-01`
- 报告导出：`python scripts/report/generate_monthly.py --user_id 1 --month 2024-01 --output report.pdf`

#### 2.4.2 任务执行流程

**同步任务流程**：
```
1. Java接收请求
2. Java通过ProcessBuilder启动Python脚本
3. Java等待进程完成（同步阻塞）
4. Python输出JSON到stdout或文件
5. Java读取结果，解析JSON
6. Java返回结果给前端
```

**异步任务流程**：
```
1. Java接收请求
2. Java创建py_job记录（status='PENDING'）
3. Java通过ProcessBuilder启动Python脚本（后台线程）
4. Java立即返回job_id给前端
5. Python执行任务，更新py_job状态（status='RUNNING' -> 'COMPLETED'）
6. 前端轮询 /api/jobs/{job_id} 获取进度和结果
7. 任务完成后，前端获取结果文件路径或摘要数据
```

#### 2.4.3 Python脚本规范

**输入输出约定**：
- 输入：命令行参数（--input input.json）或环境变量
- 输出：JSON格式（stdout或文件）
- 错误：stderr输出，返回非0退出码

**示例脚本结构**：
```python
# scripts/strategy/generate_suggestions.py
import json
import sys
import argparse

def main():
    parser = argparse.ArgumentParser()
    parser.add_argument('--user_id', type=int, required=True)
    parser.add_argument('--product_ids', type=str, required=True)  # JSON数组字符串
    parser.add_argument('--output', type=str, required=True)
    args = parser.parse_args()
    
    # 业务逻辑
    product_ids = json.loads(args.product_ids)
    suggestions = generate_suggestions(args.user_id, product_ids)
    
    # 输出结果
    with open(args.output, 'w') as f:
        json.dump(suggestions, f, ensure_ascii=False)
    
    sys.exit(0)

if __name__ == '__main__':
    main()
```

```java
public List<Suggestion> generateSuggestionsSync(Long userId, List<Long> productIds) {
    ProcessBuilder pb = new ProcessBuilder(
        "python", 
        "scripts/strategy/generate_suggestions.py",
        "--user_id", userId.toString(),
        "--product_ids", objectMapper.writeValueAsString(productIds),
        "--output", "/tmp/suggestions_" + System.currentTimeMillis() + ".json"
    );
    
    Process process = pb.start();
    int exitCode = process.waitFor();  // 同步等待
    
    if (exitCode == 0) {
        // 读取输出文件

        return objectMapper.readValue(output, new TypeReference<List<Suggestion>>() {});
    } else {
        throw new RuntimeException("Python script failed");
    }
}

public String startBacktestAsync(Long runId, BacktestInput input) {
    // 创建job记录
    PyJob job = new PyJob();
    job.setJobType("BACKTEST");
    job.setStatus("PENDING");
    job.setInputJson(objectMapper.writeValueAsString(input));
    pyJobMapper.insert(job);
    
    // 启动后台线程执行
    executorService.submit(() -> {
        try {
            job.setStatus("RUNNING");
            pyJobMapper.updateById(job);
            
            ProcessBuilder pb = new ProcessBuilder(
                "python",
                "scripts/backtest/run_backtest.py",
                "--run_id", runId.toString(),
                "--input", inputPath,
                "--output", outputPath
            );
            
            Process process = pb.start();
            int exitCode = process.waitFor();
            
            if (exitCode == 0) {
                job.setStatus("COMPLETED");
                job.setOutputJson(readOutputFile(outputPath));
                job.setArtifactPath(outputPath);
            } else {
                job.setStatus("FAILED");
                job.setError(readStderr(process));
            }
        } catch (Exception e) {
            job.setStatus("FAILED");
            job.setError(e.getMessage());
        } finally {
            pyJobMapper.updateById(job);
        }
    });
    
    return job.getJobId();
}
```

**借贷方向规则（全局统一，必须严格遵守）**：

采用方案A：`amount` 永远为正数，`posting_type` 决定方向

**资产类账户（CASH/POSITION/RECEIVABLE）**：
- `DEBIT` = 增加资产（余额增加）
- `CREDIT` = 减少资产（余额减少）
- 余额公式：`balance = initial_balance + SUM(DEBIT金额) - SUM(CREDIT金额)`

**负债类账户（LIABILITY）**：
- `DEBIT` = 减少负债（欠款减少）
- `CREDIT` = 增加负债（欠款增加）
- 余额公式：`balance = initial_balance + SUM(CREDIT金额) - SUM(DEBIT金额)`（负债余额为正数表示欠款）

**费用类账户（FEE/EXPENSE）**：
- `DEBIT` = 费用发生（费用增加）
- `CREDIT` = 费用冲减（费用减少，少见）
- 余额公式：`balance = SUM(DEBIT金额) - SUM(CREDIT金额)`（费用余额为正数表示费用总额）

**收入类账户（INCOME）**：
- `DEBIT` = 收入冲减（收入减少，少见）
- `CREDIT` = 收入发生（收入增加）
- 余额公式：`balance = SUM(CREDIT金额) - SUM(DEBIT金额)`（收入余额为正数表示收入总额）

**关键设计说明**：
- `ledger_posting` 是唯一真相源，所有余额/持仓/费用/收入均由分录计算得出
- `ledger_txn` 仅作为业务事件头，不存储金额/份额（或仅作为派生缓存，禁止手工编辑）
- UI合并展示：按 `biz_group_key` 或 `order_id` 分组，用户看到的是业务事件，不是分录
- 持仓计算：按 `product_id` 汇总 `ledger_posting` 中 `account_type='POSITION'` 的 `shares`（DEBIT增加，CREDIT减少，shares永远为正数）

**最小验算用例**：
1. 现金账户初始10000，下单买入5000：
   - 分录：CREDIT CASH 5000（现金减少）
   - 余额：10000 - 5000 = 5000 ✓
2. 转账1000：A账户减少，B账户增加：
   - 分录：CREDIT A账户 1000，DEBIT B账户 1000
   - 合计不变 ✓
3. 信用卡消费1000（负债增加）：
   - 分录：CREDIT LIABILITY 1000（负债增加），DEBIT EXPENSE 1000（费用增加）
   - 负债余额：0 + 1000 = 1000（欠款1000）✓

**快照脏数据标记与重算策略**：

1. **脏数据标记规则**：
   - 当任意历史流水/订单确认被修改（金额、份额、日期、费用等）时：
     - 标记该日期及之后所有快照为 `is_dirty=1`
     - 设置 `dirty_from_date` = 被修改的最早日期
   
2. **重算策略**：
   - 触发机制：历史修正/冲正 -> 标脏 -> 触发 `SNAPSHOT_RECALC`（可用 `py_job` 跟踪）
   - 后台定时任务：从 `dirty_from_date` 起，使用 `(T-1 快照 + T 日流水)` 逐日链式重建，直到最新
   - 性能保护：
     - >1 年前的历史修正：建议用"今日 ADJUST 分录"平账，或只允许低峰重算
     - 重算时加锁，避免并发冲突
   - 前端提示：重算进行中时，前端显示"历史修正正在回算中，请稍候"
   - 异步执行：重算任务通过 `py_job` 表跟踪，前端可轮询进度

3. **成本核算方法**：
   - 默认：`AVERAGE`（加权平均成本）
   - 可选：`FIFO`（先进先出，需启用 `position_lots` 表，Phase 2 增强）

4. **收益拆分**：
   - `realized_pnl`：从卖出/赎回的流水计算（卖出价格 - 成本）
   - `unrealized_pnl`：从持仓市值 - 成本计算
   - `income_pnl`：从分红/利息的流水汇总

## 5. 订单/结算设计详解

### 5.1 场外T+N结算流程

#### 5.1.1 买入/申购流程（关键：下单阶段是"占用/冻结"，不是"已完成扣款"）

**步骤1：用户发起买入/申购（下单阶段）**
```
1. 创建 orders 记录
   - order_id: "ORD-20240115-001"
   - status: "PENDING"
   - amount: 5000.00
   - requested_at: "2024-01-15 14:30:00"
   - trade_date: "2024-01-15" (14:30 < 15:00 cutoff)
   - expected_nav_date: "2024-01-15"
   - expected_confirm_date: "2024-01-16" (T+1)

2. 资金占用（关键！）
   - accounts.reserved_amount += 5000.00（占用增加，不直接改balance）
   - 不生成 ledger_txn 和 ledger_posting（此时还未真正扣款）

3. 可用购买力计算
   - available_for_trade = balance - reserved_amount
   - 例如：balance=10000, reserved=5000 → available=5000
```

**步骤2：待结算阶段**
```
- 首页显示"待结算清单"（orders WHERE status='PENDING'）
- 显示预期确认日期、预期净值日期、预估份额
- 显示占用金额（reserved_amount）
```

**步骤3：手工确认结算（确认阶段）**
```
1. 用户核对实际到账情况
   - 创建 settlement_confirm 记录
   - confirm_date: "2024-01-16"（可人工覆盖）
   - nav_date: "2024-01-15"（可人工覆盖）
   - confirm_nav: 1.0305（可人工覆盖）
   - confirm_shares: 4851.55（可人工覆盖）

2. 释放占用并生成真实流水
   - accounts.reserved_amount -= 5000.00（释放占用）
   - 生成 ledger_txn（txn_type='SUBSCRIPTION', status='CONFIRMED'）
   - 生成 ledger_posting（CREDIT CASH 5004.25，现金减少，实际扣款总额 = 5000 + 4.25）
   - 生成 ledger_posting（DEBIT POSITION shares=4851.55, amount=5000.00，持仓增加，成本口径不含手续费）
   - 生成 ledger_posting（DEBIT FEE amount=4.25，手续费）

3. 更新订单状态
   - orders.status = 'CONFIRMED'

4. 如有差异，生成调整分录
   - 如果预估份额 ≠ 实际份额，生成调整分录
```

**三口径公式**：
- `balance`（账面余额）= initial_balance + SUM(DEBIT) - SUM(CREDIT)
- `available_for_trade`（可用购买力）= balance - reserved_amount
- `available_for_withdraw`（可取资金）= balance - 未交收卖出款 - reserved_amount

**占用/冻结与可用资金口径（必须明确）**：
- `accounts.reserved_amount` 的含义：
  - 下单阶段：`reserved_amount += 订单金额`（占用增加，不生成流水，不扣款）
  - 结算确认：`reserved_amount -= 订单金额`（释放占用，生成真实流水）
  - 取消订单：`reserved_amount -= 订单金额`（释放占用，不生成流水）
- `available_for_trade`（可用购买力）：
  - 公式：`available_for_trade = balance - reserved_amount`
  - 含义：可用于下单的资金（已扣除占用）
- `available_for_withdraw`（可取资金）：
  - 公式：`available_for_withdraw = balance - 未交收卖出款 - reserved_amount`
  - 未交收卖出款 = SUM(赎回订单金额 WHERE status='PENDING' AND expected_confirm_date > CURDATE())
  - 含义：可以提取到银行卡的资金（已扣除占用和未到账卖出款）
  - UI必须能解释"这笔钱为何不能提现"，展示预计可取日

#### 5.1.2 赎回流程

**步骤1：用户发起赎回**
```
1. 创建 orders 记录
   - order_id: "ORD-20240115-002"
   - status: "PENDING"
   - shares: 1000.00
   - requested_at: "2024-01-15 14:30:00"
   - trade_date: "2024-01-15"
   - expected_confirm_date: "2024-01-17" (T+2)

2. 持仓减少（立即生效）
   - 生成 ledger_txn（txn_type='REDEMPTION', status='PENDING'）
   - 生成 ledger_posting（CREDIT POSITION 1000.00 shares，持仓减少）

3. 未交收卖出款记录（用于计算可取资金）
   - 记录：预计到账金额 = 1000 * 预期净值
   - 该金额在 confirm_date 之前不可取
```

**步骤2：待结算阶段**
```
- 首页显示"待结算清单"
- 显示预期确认日期、预期净值日期、预估到账金额
- 显示未交收卖出款（影响可取资金）
```

**步骤3：手工确认结算**
```
1. 用户核对实际到账情况
   - 创建 settlement_confirm 记录
   - confirm_date: "2024-01-17"（可人工覆盖）
   - nav_date: "2024-01-15"（可人工覆盖）
   - confirm_nav: 1.0305（可人工覆盖）
   - confirm_amount: 1030.50（可人工覆盖）

2. 生成到账流水
   - 更新 ledger_txn（填入实际 confirm_date、nav_date）
   - 计算成本扣减：按cost_method（AVERAGE/FIFO）计算卖出1000份对应的成本
   - 假设成本扣减 = 1000.00（示例值，实际按持仓成本计算）
   - 生成 ledger_posting（DEBIT CASH 1030.50，现金增加，实际到账净额）
   - 生成 ledger_posting（CREDIT POSITION shares=1000.00, amount=1000.00，持仓减少，成本扣减）
   - 生成 ledger_posting（DEBIT FEE amount=3.00，手续费，如有）
   - 未交收卖出款清零（现在可以取了）
   - realized_pnl = 1030.50 - 1000.00 - 3.00 = 27.50（由分录推导）

3. 更新订单状态
   - orders.status = 'CONFIRMED'
```

**未结算资金（Unsettled Cash）口径**：
- 未交收卖出款 = SUM(赎回订单金额 WHERE status='PENDING' AND confirm_date > CURDATE())
- 可取资金 = balance - 未交收卖出款 - reserved_amount
- 示例：周五卖出，T+2才可取，下周二前 available_for_withdraw 不包含该笔

### 5.2 手工结算界面设计

**待结算清单卡片**：
- 产品名称、订单类型、订单金额/份额
- 预期确认日期、预期净值日期

- 实际净值日期（可编辑，默认=预期净值日期）
- 实际确认净值（可编辑，默认=系统查询的净值）
- 实际确认份额/金额（可编辑，默认=系统计算的）
- 实际手续费（可编辑）

**关键点**：

- 订单创建：`order_id` 唯一键保证不重复
- 结算确认：`order_id` 唯一键保证一个订单只能确认一次

### 5.4 关键口径与计算公式汇总

#### 5.4.1 账户余额计算公式

**资产类账户（CASH/POSITION/RECEIVABLE）**：
```sql
balance = initial_balance + SUM(
  CASE 
    WHEN posting_type = 'DEBIT' THEN amount
    WHEN posting_type = 'CREDIT' THEN -amount
  END
)
```

**负债类账户（LIABILITY）**：
```sql
balance = initial_balance + SUM(
  CASE 
    WHEN posting_type = 'CREDIT' THEN amount  -- 负债增加
    WHEN posting_type = 'DEBIT' THEN -amount  -- 负债减少
  END
)
```

**费用类账户（FEE/EXPENSE）**：
```sql
balance = SUM(
  CASE 
    WHEN posting_type = 'DEBIT' THEN amount
    WHEN posting_type = 'CREDIT' THEN -amount
  END
)
```

**收入类账户（INCOME）**：
```sql
balance = SUM(
  CASE 
    WHEN posting_type = 'CREDIT' THEN amount
    WHEN posting_type = 'DEBIT' THEN -amount
  END
)
```

#### 5.4.2 资金口径计算公式

**可用购买力（Available for Trade）**：
```sql
available_for_trade = balance - reserved_amount
```

**未交收卖出款（Unsettled Sale Proceeds）**：
```sql
unsettled_sale_proceeds = SUM(
  CASE 
    WHEN o.order_type IN ('SELL', 'REDEMPTION') 
      AND o.status = 'PENDING' 
      AND o.expected_confirm_date > CURDATE()
    THEN COALESCE(sc.confirm_amount, o.shares * COALESCE(n.nav, 0))
    ELSE 0
  END
)
FROM orders o
LEFT JOIN settlement_confirm sc ON o.order_id = sc.order_id
LEFT JOIN nav n ON o.product_id = n.product_id AND n.nav_date = o.expected_nav_date
WHERE o.user_id = ?
```

**可取资金（Available for Withdraw）**：
```sql
available_for_withdraw = balance - unsettled_sale_proceeds - reserved_amount
```

#### 5.4.3 持仓计算

**持仓份额**：
```sql
shares = SUM(
  CASE 
    WHEN posting_type = 'DEBIT' AND account_type = 'POSITION' THEN shares
    WHEN posting_type = 'CREDIT' AND account_type = 'POSITION' THEN -shares  -- shares永远为正数，方向由posting_type决定
    ELSE 0
  END
)
FROM ledger_posting
WHERE account_type = 'POSITION' AND product_id = ?
```

**持仓成本（加权平均）**：
```sql
cost = SUM(
  CASE 
    WHEN posting_type = 'DEBIT' AND account_type = 'POSITION' THEN amount
    WHEN posting_type = 'CREDIT' AND account_type = 'POSITION' THEN -amount  -- amount永远为正数，方向由posting_type决定
    ELSE 0
  END
) / NULLIF(shares, 0)
```

#### 5.4.4 收益拆分

**已实现收益（Realized PnL）**：
```sql
-- 由postings推导：现金净流入 - 手续费 - POSITION成本扣减
realized_pnl = (
  SELECT COALESCE(SUM(CASE WHEN p.account_type = 'CASH' AND p.posting_type = 'DEBIT' THEN p.amount ELSE 0 END), 0)  -- 现金净流入
  - COALESCE(SUM(CASE WHEN p.account_type = 'FEE' AND p.posting_type = 'DEBIT' THEN p.amount ELSE 0 END), 0)  -- 手续费
  - COALESCE(SUM(CASE WHEN p.account_type = 'POSITION' AND p.posting_type = 'CREDIT' THEN p.amount ELSE 0 END), 0)  -- 成本扣减
)
FROM ledger_txn t
INNER JOIN ledger_posting p ON t.txn_id = p.txn_id
WHERE t.txn_type IN ('SELL', 'REDEMPTION') AND t.user_id = ?
```

**未实现收益（Unrealized PnL）**：
```sql
unrealized_pnl = (当前市值 - 持仓成本)
WHERE 当前市值 = 持仓份额 * 最新净值
```

**收入收益（Income PnL）**：
```sql
income_pnl = SUM(
  CASE 
    WHEN txn_type IN ('DIVIDEND_CASH', 'DIVIDEND_REINVEST', 'INTEREST')
    THEN amount
    ELSE 0
  END
)
FROM ledger_posting
WHERE account_type = 'INCOME' AND user_id = ?
```

### 5.5 分红除权 vs 派息时间错配（两步分录，避免曲线断崖）

**问题**：分红除权日（EX_DATE）市值下跌，但派息日（PAY_DATE）才到账，如果只记录派息，会导致除权日净资产凭空暴跌。

**解决方案**：使用两步分录 + 虚拟应收账户

**步骤1：除权日（EX_DATE）**
```
1. 生成 ledger_txn（txn_type='DIVIDEND_EX_DATE'）
   - trade_date: EX_DATE
   - nav_date: EX_DATE
   - note: "分红除权"

2. 生成 ledger_posting（借：应收股利 RECEIVABLE）
   - DEBIT RECEIVABLE 100.00（应收股利增加）

3. 生成 ledger_posting（贷：投资收益 INCOME）
   - CREDIT INCOME 100.00（投资收益增加）

4. 持仓市值下跌（由行情自动更新）
   - 除权后净值 = 除权前净值 - 每股分红
   - 持仓市值相应下跌

5. 净资产不变（市值下跌 + 应收增加 = 0）
   - 净资产 = 持仓市值 + 现金 + 应收股利
   - 除权日：市值-100，应收+100，净资产不变 ✓
```

**步骤2：派息日（PAY_DATE）**
```
1. 生成 ledger_txn（txn_type='DIVIDEND_PAY_DATE'）
   - trade_date: PAY_DATE
   - confirm_date: PAY_DATE
   - note: "分红到账"

2. 生成 ledger_posting（借：现金 CASH）
   - DEBIT CASH 100.00（现金增加）

3. 生成 ledger_posting（贷：应收股利 RECEIVABLE）
   - CREDIT RECEIVABLE 100.00（应收减少）

4. 净资产不变（现金增加 + 应收减少 = 0）
   - 派息日：现金+100，应收-100，净资产不变 ✓
```

**关键点**：
- 除权日：市值下跌但应收增加，净资产不崩
- 派息日：现金增加但应收减少，净资产不崩
- 使用虚拟 RECEIVABLE 账户，避免曲线断崖

### 5.5 红利再投（DIVIDEND_REINVEST）

**流程**：
```
1. 除权日：同现金分红（生成应收）
2. 派息日：不生成现金分录，直接生成持仓增加分录
   - DEBIT POSITION（份额增加 = 分红金额 / 除权后净值）
   - CREDIT RECEIVABLE（应收减少）
```

```nginx
            add_header Cache-Control "public, immutable";
        }

        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
```

## 8. 回测实验室设计详解（一等模块）

### 8.1 回测实验室定位与原则

**定位**：回测实验室是系统的"一等模块"，用于策略参数优化与验证，与生产环境完全隔离。

**核心原则**：
1. **隔离原则**：回测只读行情与历史数据，不写生产账本真相表（ledger_posting/快照等）
2. **结果存储**：摘要指标存DB，明细数据存文件（csv.gz/parquet），节省存储
3. **参数推广**：最佳参数集可一键推广到生产配置（需二次确认）
4. **成本法一致**：回测与实盘使用相同的成本核算方法（AVERAGE/FIFO）

### 6.2 回测实验室最小闭环

#### 6.2.1 BacktestPlan（回测计划）

**职责**：定义实验配置，包括：
- 产品范围（product_ids JSON数组）
- 策略版本（strategy_key + strategy_version）
- 参数空间（param_space_json，定义参数范围/枚举值）
- 时间范围（start_date, end_date）
- 初始资金（initial_cash）
- 成本法（cost_method: AVERAGE/FIFO）
- 手续费模型（fee_model_json）

**示例配置**：
```json
{
  "product_ids": [1, 2, 3],
  "strategy_key": "percentile",
  "strategy_version": "v1.0",
  "param_space": {
    "window_days": [20, 60],
    "buy_threshold": [0.2, 0.3, 0.4],
    "sell_threshold": [0.8, 0.9]
  },
  "start_date": "2023-01-01",
  "end_date": "2024-01-01",
  "initial_cash": 100000.00,
  "cost_method": "AVERAGE",
  "fee_model": {
    "buy_fee_rate": 0.000845,
    "sell_fee_rate": 0.000845,
    "min_fee": 0.20
  }
}
```

#### 6.2.2 BacktestRun（回测运行）

**职责**：执行一次回测，记录运行状态和核心指标。

**运行流程**：
1. 从 `backtest_plan` 读取配置
2. 根据 `param_space` 生成参数组合（param_set_id）
3. Python服务执行回测（读取历史行情，模拟交易）
4. 计算核心指标（总收益、年化收益、最大回撤、夏普比率、胜率等）
5. 生成明细结果文件（每日曲线、交易明细）存文件系统
6. 摘要指标存 `backtest_run` 表

**核心指标**：
- `total_return`: 总收益率
- `annual_return`: 年化收益率
- `max_drawdown`: 最大回撤
- `sharpe_ratio`: 夏普比率
- `trade_count`: 成交次数
- `win_rate`: 胜率
- `total_fees`: 手续费总额
- `fee_ratio`: 手续费占收益比例

#### 6.2.3 Compare（结果对比）

**对比维度**：
- 同 plan 下的多个 run 结果
- 按收益/回撤/波动/胜率/最大回撤期排序
- 支持多维度筛选和对比

**对比查询**：
```sql
SELECT 
  run_code,
  param_set_id,
  total_return,
  annual_return,
  max_drawdown,
  sharpe_ratio,
  win_rate,
  trade_count
FROM backtest_run
WHERE plan_id = ?
  AND status = 'COMPLETED'
ORDER BY annual_return DESC, max_drawdown ASC;
```

#### 6.2.4 Promote（参数推广）

**流程**：
1. 用户选择最佳 run（基于对比结果）
2. 系统提取 `param_set_id` 和 `param_json`
3. 生成推广预览（显示将更新的配置）
4. 用户二次确认
5. 写入 `strategy_config` 和 `product_strategy_bind`

**推广接口**：
```
POST /api/backtest/promote
{
  "runId": 123,
  "targetProductIds": [1, 2, 3],
  "confirm": true
}
```

### 6.3 回测结果存储策略

**摘要指标（存DB）**：
- 核心指标：总收益、年化收益、最大回撤、夏普比率、胜率等
- 存储位置：`backtest_run` 表

**明细数据（存文件）**：
- 每日曲线：日期、净值、持仓市值、现金余额、总资产
- 交易明细：日期、买卖方向、产品、价格、份额、金额、手续费
- 文件格式：csv.gz 或 parquet
- 存储路径：`artifact_path` 字段记录文件路径

**清理策略**：
- 结果文件TTL：默认保留90天
- 定时任务：清理过期文件
- 用户可手动清理或延长保留期

### 6.4 回测与生产隔离机制

**数据隔离**：
- 回测只读：`market_bar_daily`、`nav`、`indicator_daily`（历史数据）
- 回测不写：`ledger_posting`、`holdings_snapshot`、`net_worth_snapshot`（生产真相表）
- 回测独立表：`backtest_plan`、`backtest_run`（不影响生产）

**执行隔离**：
- Python服务独立进程执行回测（通过ProcessBuilder调用）
- 回测结果不影响生产建议生成
- 参数推广需二次确认，避免误操作

**职责边界**：
- Java职责：回测计划管理、结果对比、参数推广（需二次确认）
- Python职责：回测执行（读取历史行情，模拟交易）
- 结果存储：摘要指标存DB（`backtest_run`表），明细数据存文件（csv.gz/parquet，路径存`artifact_path`）
- 最佳参数回写：将最佳参数集写回`strategy_config`和`product_strategy_bind`（必须二次确认）

```sql
ALTER TABLE market_quote_realtime
```

- 查询策略：优先查主表，需要历史数据时再查归档表

## 10. 建议/信号模型与生命周期

### 8.1 建议生命周期状态流转

```
生成 (ACTIVE) 
  ↓
[用户操作]
  ├─→ 已执行 (EXECUTED) → 关联订单ID
  ├─→ 已忽略 (IGNORED)
  └─→ 过期 (EXPIRED) → 定时任务自动标记
```

### 8.2 去重策略

**唯一键约束**：`uk_user_product_date_strategy` (`user_id`, `product_id`, `generated_at`, `strategy_code`)

- 同产品、同策略、同日只保留最新一条建议
- `generated_at` 使用日期部分（忽略时分秒），确保同一天同一策略只生成一条
- 如果策略在同一天多次运行，新建议会覆盖旧建议（通过 `ON DUPLICATE KEY UPDATE` 实现）

### 8.3 过期管理

**过期策略**：
- `expires_at` 字段：建议有效期（默认生成后24小时）
- 定时任务：每天凌晨检查并标记过期建议
  ```sql
  UPDATE suggestions 
  SET status='EXPIRED' 
  WHERE status='ACTIVE' AND expires_at < NOW();
  ```

### 8.4 复盘最小字段集

为了支持未来复盘，建议表必须保留以下字段：

| 字段 | 用途 |
|------|------|
| `reason` | 触发理由（中文可解释） |
| `reason_blocks_json` | 结构化原因列表（用于详细分析） |
| `trigger_indicators_json` | 触发指标（分位、回撤、溢价率等） |
| `suggest_amount` / `suggest_shares` | 建议金额/份额 |
| `priority` / `confidence` | 优先级/信心度 |
| `generated_at` | 生成时间 |
| `status` / `executed_at` | 执行状态和时间 |

**复盘查询示例**：
```sql
-- 查询某产品过去30天的建议及执行情况
SELECT 
  suggestion_type, reason, priority, confidence,
  generated_at, status, executed_at, executed_order_id
FROM suggestions
WHERE user_id = ? AND product_id = ? 
  AND generated_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
ORDER BY generated_at DESC;
```

### 8.5 建议历史保留策略

- **活跃建议**：保留所有 `ACTIVE` 状态的建议
- **已执行建议**：保留至少90天（用于复盘）
- **已忽略/过期建议**：保留30天后自动清理（或仅保留摘要）

**清理任务**（可选）：
```java
@Scheduled(cron = "0 0 3 * * ?") // 每天凌晨3点
public void cleanupExpiredSuggestions() {
    // 清理30天前的已忽略/过期建议（保留摘要字段）
    LocalDate cutoffDate = LocalDate.now().minusDays(30);
    suggestionMapper.cleanupOldSuggestions(cutoffDate);
}
```

### 9.1 今日建议清单（Today Actions）

**数据来源**：`suggestions` 表

**查询条件**：
```sql
SELECT * FROM suggestions
WHERE user_id = ? 
  AND status = 'NEW'
  AND suggestion_day = CURDATE()
  AND (expires_at IS NULL OR expires_at >= NOW())
ORDER BY 
  CASE priority 
    WHEN 'HIGH' THEN 1 
    WHEN 'MEDIUM' THEN 2 
    WHEN 'LOW' THEN 3 
  END,
  generated_at DESC
LIMIT 20;
```

**展示内容（简化版）**：
- **操作内容**（核心展示）：
  - 建议类型图标 + 产品名称（如："买入 纳指ETF"）
  - 建议动作摘要（从 `action_payload` 提取，如："建议买入 5000元" 或 "建议卖出 1000份"）
  - 优先级标签（HIGH/MEDIUM/LOW）
  - 信心度（如：85%）
- **状态标识**：
  - NEW：新建议（可操作）
  - DONE：已执行（显示关联订单）
  - SNOOZE：稍后再看
  - IGNORE：已忽略
- **操作按钮**：
  - 点击操作内容：弹出完整详情（见下方）
  - 如果已下单：显示"确认成交"按钮（见下方）
  - 其他：忽略/稍后再看

**点击操作内容弹出完整详情**：
- 产品完整信息（产品代码、名称、当前净值/价格）
- 完整触发理由（`reason` 字段）
- 详细指标快照（`reason_payload` JSON展开）
- 建议动作参数（`action_payload` JSON展开：金额/目标仓位/阈值等）
- 策略信息（策略代码、版本、生成时间）
- 操作按钮：
  - 如果未下单：显示"去下单"按钮（跳转到外部交易平台或记录订单）
  - 如果已下单：显示"确认成交"按钮（见下方）

**如果已下单，在建议处进行确认/取消/修改**：
- **场景**：用户看到建议后，已在外部平台下单，现在需要在系统中录入成交信息
- **操作流程**：
  1. 点击建议的操作内容，弹出完整详情
  2. 如果检测到用户已下单（通过订单号或手动标记），显示"确认成交"按钮
  3. 点击"确认成交"，弹出成交信息录入弹窗：
     - 订单类型（买入/卖出/申购/赎回）
     - 成交价格/净值（可编辑，默认使用建议中的预期值）
     - 成交份额（买入时）或成交金额（卖出时）
     - 手续费（可编辑）
     - 成交日期（可编辑，默认今日）
     - 确认日期（T+N产品，可编辑）
     - 备注
  4. 用户确认后：
     - 创建 `orders` 记录（订单意图）
     - 如果是T+N产品：创建 `settlement_confirm` 记录（结算确认）
     - 生成 `ledger_txn` 和 `ledger_posting`（流水）
     - 更新 `suggestions.status = 'DONE'`，关联 `executed_order_id`
  5. 如果用户选择"取消"或"修改"：
     - 取消：标记建议为已忽略，不生成订单和流水
     - 修改：允许用户修改成交信息后再确认

**权限过滤**：
- 管理员：可查看全家建议（`family_id` 聚合）
- 普通成员：仅查看自己的建议

### 9.2 待结算清单（Pending Settlement）

**数据来源**：`orders` 表 + `settlement_confirm` 表（LEFT JOIN）

**查询条件**：
```sql
SELECT 
  o.*,
  pm.product_name,
  pm.product_code,
  sc.confirm_date AS actual_confirm_date
FROM orders o
INNER JOIN product_master pm ON o.product_id = pm.id
LEFT JOIN settlement_confirm sc ON o.order_id = sc.order_id
WHERE o.user_id = ? 
  AND o.status = 'PENDING'
ORDER BY o.expected_confirm_date ASC;
```

**展示内容**：
- 产品名称、订单类型、订单金额/份额
- 预期确认日期、预期净值日期
- 预估份额/金额（基于预期净值）
- 实际确认日期（如已确认）
- "确认结算"按钮

**预览净值计算**：
- 买入/申购：`预估份额 = 订单金额 / 预期净值`
- 赎回：`预估金额 = 订单份额 * 预期净值`

### 9.3 今日收益与资产概览

#### 9.3.1 总资产/净资产

**数据来源**：`net_worth_snapshot` 表（最新快照）

**查询条件**：
```sql
SELECT * FROM net_worth_snapshot
WHERE user_id = ? 
  AND snapshot_date = (
    SELECT MAX(snapshot_date) 
    FROM net_worth_snapshot 
    WHERE user_id = ?
  );
```

**展示内容**：
- 总资产、净资产、现金余额、持仓市值
- 今日盈亏 = 今日净资产 - 昨日净资产
- 累计收益 = 当前净资产 - 初始投入（从流水汇总）

#### 9.3.2 今日盈亏计算

**数据来源**：`holdings_snapshot` 表（对比昨日）

```sql
-- 今日持仓市值
SELECT SUM(market_value) FROM holdings_snapshot
WHERE user_id = ? AND snapshot_date = CURDATE();

-- 昨日持仓市值
SELECT SUM(market_value) FROM holdings_snapshot
WHERE user_id = ? AND snapshot_date = DATE_SUB(CURDATE(), INTERVAL 1 DAY);

-- 今日盈亏 = 今日市值 - 昨日市值 + 今日现金变动
```

#### 9.3.3 XIRR（资金加权收益率）计算

**数据来源**：`ledger_posting` 表（现金流）

**计算逻辑**（Python计算，Java调用）：
- 提取所有投资类交易的现金流（买入为负，卖出为正，分红为正）
- 使用 XIRR 算法计算年化收益率
- 返回结果：XIRR值、累计收益、年化收益

### 9.4 资产配置

**数据来源**：`holdings_snapshot` 表 + `accounts` 表

**按资产类别汇总**：
```sql
SELECT 
  pm.asset_type,
  SUM(hs.market_value) AS total_value,
  SUM(hs.market_value) / (SELECT SUM(market_value) FROM holdings_snapshot WHERE user_id = ?) AS weight
FROM holdings_snapshot hs
INNER JOIN product_master pm ON hs.product_id = pm.id
WHERE hs.user_id = ? AND hs.snapshot_date = CURDATE()
GROUP BY pm.asset_type;
```

**按账户汇总**：
```sql
SELECT 
  a.account_type,
  SUM(a.balance) AS total_balance
FROM accounts a
WHERE a.owner_user_id = ? OR (a.owner_family_id = ? AND a.owner_type = 'FAMILY')
GROUP BY a.account_type;
```

### 9.5 关键口径提示

#### 9.5.1 采集日 vs 净值日

**采集日（fetch_date）**：数据采集的日期（用于看板"今日资产"）

**净值日（nav_date）**：净值对应的日期（可能滞后，如QDII）

**展示策略**：
- 看板明确标注：`净值日期：2024-01-15（采集日：2024-01-16）`
- QDII产品特殊提示：`⚠️ 净值滞后1-2个交易日`
- 每个关键数字可展开查看：口径/来源/日期/计算方式

#### 9.5.2 数据来源标注

每个关键指标都应标注：
- **数据来源**：系统计算 / 行情数据 / 手工录入
- **更新日期**：最后更新时间
- **计算方式**：简要说明（如"持仓市值 = 持仓份额 × 最新净值"）

### 12.6 分红两步分录流程（避免曲线断崖）

#### 场景：基金分红，除权日2024-01-15，派息日2024-01-20，每股分红0.1元，持仓1000股

**步骤1：除权日（EX_DATE）**
```
系统处理：
1. 生成 ledger_txn
   - txn_id: "TXN-20240115-DIV-EX-001"
   - txn_type: "DIVIDEND_EX_DATE"
   - trade_date: "2024-01-15"
   - nav_date: "2024-01-15"
   - note: "分红除权"

2. 生成 ledger_posting（借：应收股利）
   - DEBIT RECEIVABLE 100.00（应收股利增加 = 1000股 × 0.1元）

3. 生成 ledger_posting（贷：投资收益）
   - CREDIT INCOME 100.00（投资收益增加）

4. 持仓市值下跌（由行情自动更新）
   - 除权后净值 = 除权前净值 - 0.1
   - 持仓市值相应下跌100元

5. 净资产不变
   - 净资产 = 持仓市值 + 现金 + 应收股利
   - 除权日：市值-100，应收+100，净资产不变 ✓
```

**步骤2：派息日（PAY_DATE）**
```
系统处理：
1. 生成 ledger_txn
   - txn_id: "TXN-20240120-DIV-PAY-001"
   - txn_type: "DIVIDEND_PAY_DATE"
   - trade_date: "2024-01-20"
   - confirm_date: "2024-01-20"
   - note: "分红到账"

2. 生成 ledger_posting（借：现金）
   - DEBIT CASH 100.00（现金增加）

3. 生成 ledger_posting（贷：应收股利）
   - CREDIT RECEIVABLE 100.00（应收减少）

4. 净资产不变
   - 派息日：现金+100，应收-100，净资产不变 ✓
```

**验证**：
- 除权日：市值下跌但应收增加，净资产不崩 ✓
- 派息日：现金增加但应收减少，净资产不崩 ✓

```sql
-- 检查资产类账户余额一致性
SELECT 
  a.id,
  a.account_code,
  a.account_type,
  a.balance AS account_balance,
  a.initial_balance + COALESCE(SUM(
    CASE 
      WHEN lp.posting_type = 'DEBIT' THEN lp.amount
      WHEN lp.posting_type = 'CREDIT' THEN -lp.amount
      ELSE 0
    END
  ), 0) AS calculated_balance,
  ABS(a.balance - (a.initial_balance + COALESCE(SUM(
    CASE 
      WHEN lp.posting_type = 'DEBIT' THEN lp.amount
      WHEN lp.posting_type = 'CREDIT' THEN -lp.amount
      ELSE 0
    END
  ), 0))) AS diff
FROM accounts a
LEFT JOIN ledger_posting lp ON a.id = lp.account_id 
  AND lp.account_type IN ('CASH', 'POSITION', 'RECEIVABLE')
WHERE a.is_active = 1 
  AND a.account_kind = 'REAL'
  AND a.account_type NOT IN ('CREDIT_CARD', 'HUABEI', 'BAITIAO', 'LOAN')
GROUP BY a.id, a.account_code, a.account_type, a.balance, a.initial_balance
HAVING diff > 0.01;  -- 差异超过1分钱报警

-- 检查负债类账户余额一致性
SELECT 
  a.id,
  a.account_code,
  a.account_type,
  a.balance AS account_balance,
  a.initial_balance + COALESCE(SUM(
    CASE 
      WHEN lp.posting_type = 'CREDIT' THEN lp.amount  -- 负债增加
      WHEN lp.posting_type = 'DEBIT' THEN -lp.amount   -- 负债减少
      ELSE 0
    END
  ), 0) AS calculated_balance,
  ABS(a.balance - (a.initial_balance + COALESCE(SUM(
    CASE 
      WHEN lp.posting_type = 'CREDIT' THEN lp.amount
      WHEN lp.posting_type = 'DEBIT' THEN -lp.amount
      ELSE 0
    END
  ), 0))) AS diff
FROM accounts a
LEFT JOIN ledger_posting lp ON a.id = lp.account_id 
  AND lp.account_type = 'LIABILITY'
WHERE a.is_active = 1 
  AND a.account_kind = 'REAL'
  AND a.account_type IN ('CREDIT_CARD', 'HUABEI', 'BAITIAO', 'LOAN')
GROUP BY a.id, a.account_code, a.account_type, a.balance, a.initial_balance
```

## 15. 验收断言（必须通过）

### 断言1：扣现金永远不会让现金余额增加
**验证**：
- 现金账户初始余额：10000
- 下单买入5000：CREDIT CASH 5000（现金减少）
- 余额计算：10000 - 5000 = 5000 ✓
- **断言通过**：扣现金操作永远导致余额减少，不会增加

### 断言2：同一业务事件 UI 可合并展示（biz_group_key）
**验证**：
- 转账操作：生成两条 ledger_txn（TRANSFER_OUT + TRANSFER_IN）
- 两条 txn 使用相同的 `biz_group_key = "TRF-20240115-001"`
- UI 查询：`SELECT * FROM ledger_txn WHERE biz_group_key = "TRF-20240115-001"`
- 展示：用户看到一条"转账"记录，而不是两条分录 ✓
- **断言通过**：UI 按 biz_group_key 合并展示，用户观感清爽

### 断言3：suggestions 同日去重成立
**验证**：
- 同用户产品、同策略版本、同日生成多条建议
- 唯一键：`uk_user_product_day_strategy` (user_product_id, strategy_version, suggestion_day)
- 新建议会覆盖旧建议（ON DUPLICATE KEY UPDATE）
- 首页查询：`WHERE status='NEW' AND suggestion_day = CURDATE()`
- 结果：同用户产品同策略版本同日只显示一条最新建议 ✓
- **断言通过**：去重键确保首页建议不爆炸

### 断言4：历史修正会触发 dirty 与重算策略
**验证**：
- 修改历史流水（2024-01-10 的流水金额）
- 系统自动标记：`holdings_snapshot.is_dirty = 1`，`dirty_from_date = '2024-01-10'`
- 后台定时任务：从 2024-01-10 起逐日链式重建快照
- 重建逻辑：T 日快照 = (T-1 日快照 + T 日流水)
- 结果：所有受影响日期的快照都被正确重建 ✓
- **断言通过**：历史修正触发脏数据标记和级联重算

### 断言5：可取资金不会包含未交收卖出款
**验证**：
- 周五卖出基金，T+2 才可取（下周二到账）
- 订单状态：PENDING，confirm_date = 下周二
- 未交收卖出款 = SUM(赎回订单金额 WHERE status='PENDING' AND confirm_date > CURDATE())
- 可取资金 = balance - 未交收卖出款 - reserved_amount
- 结果：下周二前，可取资金不包含该笔卖出款 ✓
- **断言通过**：可取资金口径正确，不会包含未交收资金

### 断言6：分红除权日净资产不崩
**验证**：
- 除权日：市值下跌100元，应收股利增加100元
- 净资产 = 持仓市值 + 现金 + 应收股利
- 除权日：市值-100，应收+100，净资产不变 ✓
- 派息日：现金+100，应收-100，净资产不变 ✓
- **断言通过**：分红两步分录确保曲线不崩

### 断言7：账户余额与流水严格一致
**验证**：
- 对账SQL：`balance = initial_balance + SUM(DEBIT) - SUM(CREDIT)`（资产类账户）
- 定时任务每天检查：`ABS(balance - calculated_balance) > 0.01` 报警
- 结果：所有账户余额与流水计算值一致 ✓
- **断言通过**：余额与流水严格一致，可追溯

### 断言8：下单占用不影响余额，结算确认才扣款
**验证**：
- 下单：reserved_amount += 5000，balance 不变
- 可用购买力：available = balance - reserved = 10000 - 5000 = 5000 ✓
- 结算确认：reserved_amount -= 5000，生成 CREDIT CASH 5000，balance -= 5000
- 结果：下单阶段不扣款，结算确认才扣款 ✓
- **断言通过**：占用机制正确，不会提前扣款

## 17. 最小验算用例（必须通过）

### 用例1：扣现金永远不会让现金余额增加

**初始状态**：
- 现金账户：balance = 10000, reserved = 0

**操作1：下单买入5000元（占用阶段）**
- reserved += 5000
- balance 不变
- **验证**：balance = 10000, reserved = 5000, available = 5000 ✓

**操作2：结算确认（扣款阶段）**
- 生成 ledger_posting：CREDIT CASH 5000（现金减少）
- reserved -= 5000
- balance -= 5000
- **验证**：balance = 5000, reserved = 0, available = 5000 ✓

**断言**：扣现金操作永远导致余额减少，不会增加 ✓

### 用例2：转账账户合计不变

**初始状态**：
- 账户A：balance = 10000
- 账户B：balance = 5000

**操作：转账1000元（A → B）**
- 生成 ledger_posting：CREDIT A账户 1000（A减少）
- 生成 ledger_posting：DEBIT B账户 1000（B增加）
- **验证**：
  - 账户A：balance = 10000 - 1000 = 9000 ✓
  - 账户B：balance = 5000 + 1000 = 6000 ✓
  - 合计：9000 + 6000 = 15000（不变）✓

**断言**：转账操作账户合计不变 ✓

### 用例3：信用卡消费负债增加

**初始状态**：
- 信用卡账户（LIABILITY）：balance = 0（无欠款）

**操作：信用卡消费1000元**
- 生成 ledger_posting：CREDIT LIABILITY 1000（负债增加）
- 生成 ledger_posting：DEBIT EXPENSE 1000（费用增加）
- **验证**：
  - 负债余额：0 + 1000 = 1000（欠款1000）✓
  - 费用余额：0 + 1000 = 1000（费用1000）✓

**操作：还款333.33元（本金）**
- 生成 ledger_posting：DEBIT LIABILITY 333.33（负债减少）
- 生成 ledger_posting：CREDIT CASH 333.33（现金减少）
- **验证**：
  - 负债余额：1000 - 333.33 = 666.67（剩余欠款）✓

**断言**：信用卡消费和还款分录正确，负债余额计算准确 ✓

### 用例4：分红除权日净资产不崩

**初始状态**：
- 持仓市值：10000
- 现金：0
- 应收股利：0
- 净资产：10000

**操作：除权日（EX_DATE）**
- 持仓市值下跌：10000 - 100 = 9900（除权）
- 生成 ledger_posting：DEBIT RECEIVABLE 100（应收增加）
- 生成 ledger_posting：CREDIT INCOME 100（收入增加）
- **验证**：
  - 净资产 = 持仓市值 + 现金 + 应收 = 9900 + 0 + 100 = 10000（不变）✓

**操作：派息日（PAY_DATE）**
- 生成 ledger_posting：DEBIT CASH 100（现金增加）
- 生成 ledger_posting：CREDIT RECEIVABLE 100（应收减少）
- **验证**：
  - 净资产 = 持仓市值 + 现金 + 应收 = 9900 + 100 + 0 = 10000（不变）✓

**断言**：分红两步分录确保净资产不崩 ✓

### 用例5：可用购买力与可取资金口径

**初始状态**：
- 现金账户：balance = 10000, reserved = 0
- 未交收卖出款：0

**操作1：下单买入5000元**

- **验证**：

- **验证**：

  - available_for_withdraw = 10000 - 未交收卖出款 - 5000（不包含卖出款）✓
  - UI显示："预计下周二可取" ✓

**断言**：资金口径正确，用户可理解 ✓

## 附录：命名规范建议

### 表名规范
- 使用小写字母 + 下划线：`ledger_txn`, `product_master`
- 复数形式：`users`, `accounts`, `orders`
- 单数形式（主数据）：`product_master`, `user_product`

### 字段名规范
- 使用小写字母 + 下划线：`user_id`, `product_code`, `confirm_date`
- 布尔字段：`is_active`, `is_manual_override`
- 日期字段：`created_at`, `updated_at`, `trade_date`, `nav_date`

### 接口路径规范
- RESTful风格：`/api/{resource}/{id}`
- 动作使用动词：`/api/orders/{id}/cancel`, `/api/settlements/confirm`
- 查询使用查询参数：`?status=&startDate=&endDate=`

### 枚举值规范
- 使用大写字母 + 下划线：`PENDING`, `CONFIRMED`, `ACTIVE`, `EXECUTED`
- 类型枚举：`BUY`, `SELL`, `SUBSCRIPTION`, `REDEMPTION`
